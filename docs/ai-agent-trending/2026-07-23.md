## 今日 Feature 趋势摘要

1. **AI/Agent 仍占据高位，但形态更分散。** 19 个席位中有 11 个直接相关，覆盖实时情报、WiFi 感知、响应 Skill、语音工作室、模型网关、金融基础模型、Skill 目录、Coding Agent UI、AI 教材、代码上下文与结构化输出。
2. **“给 Agent 补上下文/能力”比再做一个聊天框更热。** World Monitor 提供实时世界数据，Code Review Graph 缩小代码阅读面，awesome-claude-skills 提供工作流资产，Outlines 约束机器可消费的输出；Agent 产品正在围绕数据、工具、协议和结果可靠性扩展。
3. **Local-first 同时出现在隐私、成本与硬件三条线上。** Voicebox 本地完成语音输入输出，RuView 把感知留在边缘侧，World Monitor 可接 Ollama，Code Review Graph 本地保存代码图；但“本地”不等于零模型下载、零硬件、零联网数据源或零配置。
4. **开箱即用承诺必须拆成层级。** i-have-adhd 和 pi-web 接近一条安装命令即可体验；Voicebox 需要下载模型并受 GPU/平台影响；Kronos 需要准备 K 线、Python 环境和模型；RuView 的真实感知需要 ESP32 等硬件，Docker 模拟不能证明真实环境效果。
5. **可视 Demo 与极低试用摩擦是共同传播器。** 今日 #1 World Monitor 有高密度地图，RuView 有“WiFi 穿墙感知”强叙事，Voicebox 有完整桌面 UI，Kronos 有 BTC 预测页，pi-web 直接给现有 CLI 会话加浏览器工作台。
6. **热度与成熟度依然错位。** 榜单同时包含无 Release 的 Skill/清单/研究模型、0.x 高频迭代应用、1.x SDK 和一天内刚发版的工具；Stars 暴增、README 功能表和生产可靠性不是同一件事。

### 2. [RuView](https://github.com/ruvnet/RuView)

**典型场景：** 无摄像头灯光/HVAC/会议室占用；老人照护、睡眠和跌倒提醒原型；WiFi 姿态/活动研究；Agent 查询物理环境；多节点穿墙与遮挡实验。

| **结构 + 可选向量搜索** | FTS 找入口，可接本地/云 embedding 再沿关系遍历 → 概念发现兼顾结构事实 → 不知符号名时 | 云 embedding 可能外发摘要/收费；需显式安装配置，[FAQ](https://github.com/tirth8205/code-review-graph/blob/main/docs/FAQ.md#isnt-this-just-rag) |
| **执行流 / 社区 / 健康** | 找关键链、hub/bridge、意外耦合、孤立/未测热点 → 明确阅读与债务优先级 → Onboarding/架构体检 | JS/Go 流检测 recall 较弱 |
| **可视化 / 导出 / Wiki** | D3 图和 JSON/GraphML/SVG/Cypher/Obsidian/Wiki → 人可核查与复用 → 架构沟通 | 可能含绝对路径/结构元数据，公开前清理 |
| **VS Code 面板** | Graph、Blast Radius、Review、callers/callees/tests、搜索与自动更新 → 不离编辑器探索 → VS Code | 依赖 Python CLI且分开发版；有 native binary 激活问题报告 |

**易用性与同类差异：** 核心 SQLite 图、CLI、MCP、关键词搜索、可视化、watch/daemon 不需云服务；一键配置多个 Agent，并有多语 README、示例和复现实验。Serena/LSP 在单语言类型/重载/诊断/安全重命名更强，CRG 强在跨语言持久 Review 图；Repomix 适合小仓一次性打包，CRG 适合重复多跳；向量 RAG 擅长概念，CRG 擅长调用/影响/测试。

**限制与关注原因：** “82× 中位/528× 最高”比较图查询与读整个语料库，不是熟练 Agent 的 grep+top-files；评测 runner 尚有未自动建向量索引问题。动态分派、元编程、同名/重载和跨语言调用可能误连；关键词 MRR 0.35、流程 recall 33%，小仓一次性问题未必划算；11,872 文件首建曾有 45 分钟报告。事实：今日 #14、+882，最新 [v2.3.7](https://github.com/tirth8205/code-review-graph/releases/tag/v2.3.7) 集中强化 CLI、框架覆盖、影响/测试与 Action。推断：上下文成本痛点、一键多平台和醒目的 token-savings 数字共同推动热度，但价值应由真实多跳 Review 的准确性验证。

### 11. [Outlines](https://github.com/dottxt-ai/outlines)

**一句话定位：** 面向要把 LLM 输出可靠接入程序的 Python 开发者，用类型或语法在生成时限定输出结构，减少坏 JSON、脆弱后处理和格式重试。

**典型场景：** 客服/合同/商品抽取入库；固定标签分类；Agent 工具参数生成；日期/ID 格式；自定义 DSL、表达式或受控命令。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **Python types / Choice** | 传 int/float/bool、Literal/Enum、集合、Union/Optional 或动态 Choice → 复用类型提示、避免标签漂移 → 分类/数值抽取 | 开源可用，[Output Types](https://dottxt-ai.github.io/outlines/latest/features/core/output_types/) |
| **Pydantic / JSON Schema / 函数签名** | 接 Pydantic、dataclass、TypedDict、GenSON、Callable/原始 Schema → 复用数据契约 → 抽取/表单/工具参数 | 只生成参数，不执行工具；v1 返回字符串，仍需解析/验证 |
| **Regex / Regex DSL** | 强制匹配模式，提供 email、ISBN、sentence 等组合 → 一次得格式合规结果 → ID/日期/协议字段 | 保证形式不保证业务事实，[文档](https://dottxt-ai.github.io/outlines/latest/features/utility/regex_dsl/) |
| **CFG** | 用 Lark 文法限定上下文无关语言 → JSON 外也可保证句法 → 算式/DSL/受控代码 | 相对原生 Structured Outputs 的强项，也是兼容/性能风险最高能力 |
| **统一模型/Provider 接口** | 适配 Anthropic、Gemini、llama.cpp、LM Studio、MLX、Mistral、Ollama、OpenAI、SGLang、TGI、Transformers、vLLM 等 → 同一调用切后端 → 评测/迁移 | “any model”不等于每家支持所有约束，[矩阵](https://dottxt-ai.github.io/outlines/latest/features/models/) |
| **Text / Chat / Multimodal 输入** | 统一 Chat，并以 Image/Video/Audio 表达资产 → 结构化抽取扩到图片等 → 票据/截图 | 模态能力来自底层模型，[Inputs](https://dottxt-ai.github.io/outlines/latest/features/core/inputs/) |
| **Generator / Application** | 固定模型与输出契约，把模板/类型封成可换模型组件 → 少重复配置 → 固定抽取/分类 | 不提供可视工作流或 Agent Runtime，[Application](https://dottxt-ai.github.io/outlines/latest/features/utility/application/) |
| **Jinja Templates** | 支持字符串/文件、嵌套和 filter → Prompt 与业务代码分离 → few-shot/团队维护 | 开源可用，[Template](https://dottxt-ai.github.io/outlines/latest/features/utility/template/) |
| **Batch / Streaming / Async** | 改善离线吞吐或首字体验 → 批量抽取/在线应用 | 非统一保证；batch 只在少数后端可用 |

**易用性与同类差异：** `pip install outlines`，Python API 短，有逐 Provider 矩阵和示例；无独立 UI，也不是 Agent 框架。相较 Instructor，Outlines 强在生成期硬约束、Regex/CFG 和本地后端，弱在直接返回 typed object、业务 validator、失败反馈重试和 partial/iterable streaming；相较 Guidance，它接口更直接，弱在将控制流/选择/捕获穿插进一次生成；相较 Provider 原生能力，它统一类型和本地语法约束更强，但远程仍受各家 JSON Schema 子集限制。

**状态、限制与关注原因：** 必须另备本地模型或云 Key/额度；`.txt API` 仍 early access，只支持有限 JSON Schema，XML/FHIR/custom grammar 是邀约入口。结构有效不等于语义正确，复杂 Schema 可能慢/被拒，`max_new_tokens` 截断也会破坏结果；缓存只减少重复准备约束，不是结果缓存。[1.3.2](https://github.com/dottxt-ai/outlines/releases/tag/1.3.2) 于 7 月 20 日修复 Schema、日期、Union、vLLM、LM Studio 等兼容问题，说明维护活跃也说明组合面仍需回归。事实：今日 #19、+364。推断：Agent 工具调用/数据抽取的机器可消费输出刚需、Provider 碎片化和近期修复版共同推动热度。

## 今天最值得体验的 3 个项目

1. **[World Monitor](https://github.com/koala73/worldmonitor)：先体验“数据—可视化—Agent”是否真能少切站。** 最吸引人的不是单个 AI Chat，而是免登录地图、带来源 Brief、国家风险、航运/基础设施图层和 MCP/API 同处一个产品。先用 Free Dashboard/国家 Brief，再决定 Route/Scenario/MCP 是否值得付费；不要把 AI 摘要当作已核验情报。
2. **[Voicebox](https://github.com/jamiepine/voicebox)：体验本地双向语音闭环。** 它把声音克隆、多引擎 TTS、Whisper 听写、Captures、Stories、persona 与 Agent MCP 发声装进一个 UI，Feature 组合比单点 TTS 更有产品感。优先用预设声音和短听写测试硬件/权限，确认稳定后再导入真实声音；远程 API 不要直接暴露公网。
3. **[pi-web](https://github.com/agegr/pi-web)：Pi 用户最低摩擦的可视升级。** 一条 `npx` 即可获得会话浏览/续聊/分叉、worktree、文件/图片/PDF 预览、上下文/成本、模型和 Skill 管理；它不替换 Pi Runtime，而是把已有本地会话变成更易恢复的工作台。远程访问前先限制监听地址并加自己的网络边界。

## 值得持续跟踪的新 Feature 方向

1. **物理世界通过 MCP 进入 Agent。** RuView 把 presence/vitals/pose 暴露为工具，说明 Agent Grounding 正从网页/数据库走向传感器；下一阶段关键不是工具数量，而是 source 标记、标定、置信度、ground truth 和安全降级。
2. **本地双向语音从“模型 Demo”走向工作台。** Voicebox 将 STT、TTS、声音资产、persona、时间线和 Agent 发声闭环；值得跟踪流式转写、跨平台自动粘贴、低延迟打断、声音授权证明和远程 API 鉴权。
3. **Agent 上下文进入“结构化预算”阶段。** Code Review Graph 用代码关系选择上下文，OmniRoute 压缩工具输出，Outlines 约束输出结构；未来 Feature 应同时公开 token 节省、召回/语义损失和失败恢复，而不是只给压缩百分比。
4. **Skill 生态需要供应链与兼容治理。** i-have-adhd 和 awesome-claude-skills 证明可安装工作流极易传播；下一步需要版本锁定、签名/权限声明、宿主兼容测试、可复现评测和自动卸载/回滚。
5. **领域基础模型需要从论文任务走到可校准产品。** Kronos 的金融 OHLCVA 生成很直观，但真正有价值的后续是已知协变量、概率校准、滚动/成本后回测、稳定服务和对未开放大模型成绩的严格分层。
6. **多模型网关将从“能接多少家”转向合规与质量治理。** OmniRoute 展示了额度、预算、容灾、压缩和 Agent 自管理；持续跟踪点是 Provider ToS、OAuth/订阅转发边界、质量回归、统一结算与企业支持，而不是目录数字继续膨胀。
