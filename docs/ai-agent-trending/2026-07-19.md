## 今日 Feature 趋势摘要

1. **AI 热点从 LLM 向“空间智能 + 低资源推理”外扩。** 榜首 LingBot-Map 把长视频变为相机轨迹与点云场景；AirLLM 则继续用“有限显存运行超大模型”吸引本地推理用户。
2. **Agent 的竞争焦点越来越像“谁能提供更好的上下文”。** Code Review Graph 提供代码关系图，Wigolo 提供本地 Web 检索，Ossie 提供统一业务语义，PostHog 提供真实产品行为、错误与收入信号。
3. **MCP 已从单一工具连接器变成产品入口。** 今天既有面向编码 Agent 的本地 Web MCP，也有代码知识图 MCP、PostHog 大范围产品工具 MCP；价值从“能调用工具”转向“最小权限、可过滤、可复现”。
4. **Agent Skills/Harness 更加垂直。** UI Skills 把设计工程规则做成可发现、可路由的 Skill 集；Kimi CLI 则把 coding agent、Agent Client Protocol、MCP、Skills 和自动化入口集中到一个 CLI。
5. **本地优先仍是明显卖点，但含义各不相同。** 有的只把索引留在本地，有的仍会调用外部模型或搜索服务；“无云”“无 API Key”“离线”不能互相替代，报告逐项标出数据是否可能出设备。
6. **热度与成熟度错位依旧突出。** 今天有 Draft 规范、Public/Open Beta、无正式 Release 的研究仓库、依赖第三方模型的教程，以及作者自报性能数字；“可运行 Demo”不能自动升级为生产 SLA。

## Feature 深度卡片

## 1. LingBot-Map

**一句话定位：** 面向机器人、空间智能和 3D 视觉研究者，把连续 RGB 图片或视频转换为相机轨迹、深度/三维点及可浏览点云，重点解决传统多视图重建难以持续处理长视频的问题；它不是 LLM、Agent 框架或带回环/全局优化的完整生产 SLAM。[README](https://github.com/Robbyant/lingbot-map#readme) · [论文](https://arxiv.org/abs/2604.14141)

**典型场景：** 长程机器人/车载/航拍建图原型；室内外 walkthrough 点云预览；把数千至 25,000 帧视频离线重建成展示片；检查世界模型生成视频的几何一致性；在 KITTI、ETH3D、TUM 等数据集做轨迹、深度和点云评测。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **流式 RGB→3D 与相机轨迹** | 连续接收图片并逐帧产出位姿和三维信息 → 不必等整段视频到齐 → 在线感知原型、长视频处理 | **已可用**；[定义与输出](https://arxiv.org/abs/2604.14141) |
| **万帧级连续推理** | 作者报告 518×378 下约 20 FPS、可处理 >10k 帧 → 把重建从照片集推向真实长程视频 → 巡检、道路、航拍 | 能力与 Demo 已提供；**20 FPS/SOTA 是特定条件下的作者报告，不是 SLA** |
| **Keyframe Interval** | 只把每 N 帧之一留作上下文，其他帧仍预测 → 用少量质量换显存和长度 → 高帧率、显存紧张序列 | **已可用，需调参**；[Keyframe](https://github.com/Robbyant/lingbot-map#streaming-with-keyframe-interval) |
| **Windowed Inference** | 将 >3,000 帧素材分窗，保留跨窗重叠以改善对齐 → 超出训练尺度仍有明确降级路径 → 数千/数万帧 | **已可用**；不能等同全局回环/位姿图优化。[窗口模式](https://github.com/Robbyant/lingbot-map#windowed-inference-for-long-sequences-3000-frames) |
| **三种 Checkpoint** | `long` 偏长序列、标准版平衡、`stage1` 可载入 VGGT 做双向推理 → 按任务选权重 → 长程/通用/兼容实验 | 可下载，单个约 4.6–4.8GB；[模型说明](https://github.com/Robbyant/lingbot-map#model-download) |
| **本地 3D Viewer** | `demo.py` 启动 Viser，可调置信度、点大小、下采样 → 一条命令看到点云并排错 → 小中型场景体验 | **已交付**；需视觉 extra、权重和本地浏览器。[Quick Start](https://github.com/Robbyant/lingbot-map#quick-start) |
| **离线点云成片** | `batch_demo.py` 从视频/图片完成推理并无头生成 flythrough MP4 → 省去手拼推理、点云与编码 → 大场景/服务器展示 | **已交付但安装较重**；需 Open3D、Kaolin、FFmpeg、CUDA 扩展。[离线管线](https://github.com/Robbyant/lingbot-map#offline-rendering-pipeline-demo_renderbatch_demopy) |
| **YAML 虚拟镜头** | 编排 follow/birdeye/static/pivot 与过渡 → 不改代码即可生成不同展示视角 → Demo、汇报 | 已提供三套 preset；[Camera Path](https://github.com/Robbyant/lingbot-map#camera-path-yaml) |
| **天空移除** | 用 ONNX 分割过滤室外天空点，缓存 mask → 减少漂浮点并避免重复成本 → 道路、校园、航拍 | 需 ONNX Runtime 和首次外部下载；[Sky Masking](https://github.com/Robbyant/lingbot-map#sky-masking) |
| **显存/速度/精度开关** | CPU offload、减少 scale frames/refinement、SDPA、图像/点云下采样 → 在不同硬件上先跑起来 → 消费级 GPU | 已提供，但**未承诺 4GB 可跑**；已有 [4GB OOM 报告](https://github.com/Robbyant/lingbot-map/issues/73) |
| **可复用中间产物** | 保存逐帧 NPZ、完整 YAML、RGB 对照与耗时 → 不重新推理即可换镜头并复现实验 → 多版成片、团队复核 | 已可用；[输出文件](https://github.com/Robbyant/lingbot-map#output-files) |
| **多数据集 Benchmark/Viewer** | 统一跑轨迹、位姿、深度、点云指标并可视对比 → 从“漂亮 Demo”回到量化回归 → 学术复现 | 需自行下载数据与额外环境；[Benchmark](https://github.com/Robbyant/lingbot-map/tree/main/benchmark) |

**开箱程度：** 权重、四个小例子、Viewer、长视频参数、离线渲染和 Benchmark 均已交付；但“开箱”仍要求 Python 3.10、PyTorch/CUDA、本地 NVIDIA GPU和大文件下载。天空、渲染、Benchmark 各有额外依赖。未发现必须付费服务。

**同类差异：** 相比 [StreamVGGT](https://github.com/wzzheng/StreamVGGT)，LingBot-Map 更完整地交付 >10k/25k 帧降级策略和点云成片，但缺 4D 动态输出、训练代码与托管 Demo；相比 [VGGT](https://github.com/facebookresearch/vggt)，长视频工作流更直接，但缺单视图、COLMAP、bundle adjustment、gsplat/3DGS 下游和训练/微调。

**易用性与限制：** 有本地 UI、脚本 CLI、YAML preset、可下载 Demo、详细 README 和 Benchmark；无稳定 SDK/API/服务端、独立文档站、正式 Release 或训练代码。320 views 左右是重要训练上下文边界；窗口策略不等于无限程全局一致；未核实到 loop closure、COLMAP/Nerfstudio、NeRF/3DGS、mesh/纹理资产导出。动态视频输入也不能外推为独立动态物体/4D 建模。

**为何今天关注：** 事实是它位列今日第 1、+831 Stars，并公开权重、25k 帧实例、完整命令和多数据集评测。[README News](https://github.com/Robbyant/lingbot-map#news) 推断上，“13 分钟视频→可飞行浏览点云”的视觉传播力、空间智能/机器人/世界模型热点和近期文档补全共同放大了热度。

## 2. Apache Ossie

**一句话定位：** 面向数据平台、BI/AI 工具厂商和数据团队，提供厂商中立的 JSON/YAML 语义模型交换规范，让指标、字段、关系和业务提示在不同工具及分析 Agent 间保持一致；它为 Agent 提供 grounding，但不运行模型、Agent 或查询。[README](https://github.com/apache/ossie#readme) · [Core Spec](https://github.com/apache/ossie/blob/main/core-spec/spec.md)

**典型场景：** 给自然语言数据 Agent 提供认证指标与业务同义词；在 dbt/GoodData/Omni/Polaris/Salesforce 等格式间迁移；用 Git 统一 KPI；为工具厂商实现导入/导出；在 CI 校验模型。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **中立 JSON/YAML 规范** | 用统一文档和机器 Schema 描述语义模型 → 减少供应商锁定、便于 Git Review → 多 BI/AI 共存 | 可写可读；`main` 为 **`0.2.0.dev0` Draft**，需锁 commit。[Core Spec](https://github.com/apache/ossie/tree/main/core-spec) |
| **Dataset/Field/Relationship/Metric** | 同时携带来源、键、字段用途、关系和指标表达式 → Agent 不再从裸表名猜业务 → 数据字典、指标问答 | 核心可用；grain、复杂关系、层级等仍薄 |
| **复合键与跨数据集关系** | 用成对列数组表示 Join → 降低 Agent 猜关联条件风险 → 订单-客户、跨表 KPI | 已可表达；缺一等 cardinality/grain 等完整运行语义 |
| **多层级 `ai_context`** | 在模型、数据集、字段、关系、指标写 instructions/synonyms/examples → 把“GMV=销售额”等提示随模型流转 → Text-to-SQL、指标发现 | 字段已可用；**verified queries、可见性与安全控制仍是路线图** |
| **多方言表达式** | 同时携带 ANSI、Snowflake、Databricks、BigQuery、Tableau、MDX、MAQL 等 → 避免强压成一个方言 → 多仓库迁移 | 可用；不是自动 SQL 翻译器。[方言](https://github.com/apache/ossie/blob/main/core-spec/spec.md#dialects) |
| **`custom_extensions`** | 保存核心规范不能表达的厂商 JSON → 提高回转保真度 → 渐进迁移 | 可用；其他消费者通常只能透传，不能理解或执行 |
| **参考转换器** | 覆盖 dbt、GoodData、Honeydew、Omni、Orionbelt、Polaris、Salesforce，及 Ossie→Snowflake → 降低集成 PoC 成本 → 迁移/兼容测试 | 源码可试；方向、安装和保真度不一。[Converters](https://github.com/apache/ossie/tree/main/converters) |
| **独立 Validator** | 检查 Schema、重名、关系引用及部分 SQL 语法 → 把明显错误前移 CI → 模型治理 | 需 Python 3.11+；MDX/Tableau/MAQL 跳过语法验证。[Validator](https://github.com/apache/ossie/blob/main/validation/validate.py) |
| **Python 类型模型** | 用 Pydantic 读取、校验和生成文档 → 转换器/API 开发获得类型提示 → Python 数据工具 | 源码可用、版本同为 dev；未见稳定包/Release |
| **TPC-DS/Flights 示例与 Fixtures** | 提供真实规模模型和转换测试样本 → 不只从玩具 Schema 起步 → PoC、回归 | 开箱可读/可跑；无托管 UI/Demo。[Examples](https://github.com/apache/ossie/tree/main/examples) |

**开箱程度：** 规范、Schema、样例、验证脚本和各转换器已存在；转换器可能需要 dbt manifest、Java/Maven、平台 endpoint/OAuth 或商业数据产品账号。Ossie 本身无付费层。统一 `ossie convert/validate/plugin` CLI 当前只是返回 `not yet implemented` 的脚手架，不能写成可用产品。

**同类差异：** 相比 dbt MetricFlow，Ossie 强在厂商中立、扩展透传、多方言与跨生态转换，但弱在可执行指标、Join 规划和 SQL 生成；相比 Cube，它不要求采用同一运行产品，却缺 SQL/REST/GraphQL/MCP、缓存、权限、UI 和查询 Serving。Ossie 适合交换合同，不足以让 Agent 单独查询数据。[Roadmap](https://github.com/apache/ossie/blob/main/ROADMAP.md)

**易用性与限制：** JSON/YAML、Python 类型、Validator、Converters 与示例齐全，但无 UI、网络 API、稳定统一 CLI 或正式 Release。“Write once, query anywhere”目前更准确是“write once, exchange with adapters”；AI Context 只提供更好提示，不保证正确性、权限或认证答案；转换器也不普遍无损。

**为何今天关注：** 事实是项目在 6 月进入 Apache Incubator、7 月完成 OSI→Ossie 更名，dbt Core 1.12 开始接受 Ossie 文档，本周又合入 Omni 转换器和 CLI 脚手架。[Apache 公告](https://ossie.apache.org/updates/ossie-enters-apache-incubator/) · [dbt 说明](https://docs.getdbt.com/docs/build/about-metricflow) 推断上，Apache 中立治理和“给分析 Agent 统一业务语义”需求共同推动关注。

## 3. PostHog

**一句话定位：** 面向产品工程和 AI 应用团队，把产品分析、Session Replay、错误、日志、Flags、实验与 LLM/Agent traces 放进同一上下文，再让 Web AI、MCP 或 Self-driving Agent 调研问题并产出**需人工审核合并**的 PR；它不是已经能完全自治经营产品的 Agent。[README](https://github.com/PostHog/posthog#posthog-is-the-open-source-platform-for-building-self-driving-products) · [Self-driving](https://posthog.com/docs/self-driving)

**典型场景：** 从 rage click/错误/收入影响定位并修复问题；监控 Agent 成本、延迟、工具和质量；让 Coding Agent 通过 MCP 带产品上下文工作；自然语言分析与建仪表盘；在 Slack 发起数据/代码任务。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **Context Warehouse / Product OS** | 联合事件、Replay、错误、问卷、Flags、实验、仓库与业务数据 → Agent 同时看“发生什么、影响谁、值不值得修” → 排障/优先级 | 核心平台可用；价值取决于埋点和数据源 |
| **PostHog AI** | Web 内自然语言研究、SQL、Insight/Dashboard、Replay/实验总结、建 Flag/问卷 → 降低 HogQL/UI 门槛 → PM/分析师/工程师 | Cloud 内置；需 AI data processing，超免费 credits 计费。[能力页](https://posthog.com/docs/posthog-ai) |
| **Hosted MCP** | 让 Codex/Claude/Cursor 查询 analytics、错误、日志、Flags、实验、Prompts、AI traces，并可创建/更新对象 → 从诊断走到行动 → 编辑器内排障/实验 | endpoint 免费，底层用量/AI 工具仍计费；[MCP](https://posthog.com/docs/model-context-protocol) |
| **MCP 最小权限** | 固定 org/project、read-only、按 Feature/工具过滤，提供 token-efficient CLI mode → 缩小写操作面和 Schema 体积 → 生产只读接入 | 已可用；官方仍警告 Prompt Injection。[FAQ](https://posthog.com/docs/model-context-protocol/faq) |
| **AI Observability** | 捕获 generation/trace/span/tool、token、成本、延迟、错误并关联用户、Replay、漏斗 → 同时看模型性能与产品结果 → RAG、Agent、多轮工具链 | 需 wrapper/OTel/手工 `$ai_` 事件；Cloud 有免费额度。[Start](https://posthog.com/docs/ai-observability/start-here) |
| **AI Evals** | LLM Judge、Hog 确定性规则、本地 sentiment，支持模板、采样、报告和通知 → 抽查变持续质量信号 → 幻觉/安全/格式/情绪 | Judge 需 Provider Key/模型费；Eval 仍计 AI Event。[Evals](https://posthog.com/docs/ai-evals) |
| **Prompt Management + Playground** | 变量、不可变版本、Diff/恢复、SDK 缓存；多模型并排、Trace Replay → 生产失败可复现并沉淀版本 → PromptOps | Prompt Management **Beta**；完整模型需 BYOK。[Prompt](https://posthog.com/docs/prompt-management) |
| **Scouts + Signals + Reports** | 定时读取错误、Replay、支持、Jira/Linear 等，跨运行记忆并去重聚合证据 → 从等人报 Bug 变主动发现 → 维护/持续优化 | **Self-driving Open Beta**；结论是 Agent 判断，不是确定告警。[Scouts](https://posthog.com/docs/self-driving/scouts) |
| **Human-gated PR Loop** | 在沙箱核实数据/代码、开 PR、跑 CI、回应 Review，合并后再验证指标 → 把告警压成可审工件 → 小步修复 | Open Beta；不能自 merge；前 3 PR/月免费，之后按 PR 计费。[闭环](https://posthog.com/docs/self-driving/self-improving-loop) |
| **Slack Agent** | 在 Channel/Thread 中查数据、启动沙箱任务、开 Draft PR，多人追加约束 → 把发现与交付留在协作流 → 支持转修复 | **Beta**；需 Slack/GitHub OAuth、计 AI credits，有沙箱/超时限制。[Slack](https://posthog.com/docs/slack) |
| **CLI / PostHog Code** | CLI 可脚本化 SQL、MCP、任务、CI；Code 文档展示本地/worktree/Cloud 并行 Agent 与 Diff/PR Review → 适合自动化和长任务 | CLI 已可装；Code 源码公开，但[下载页](https://posthog.com/docs/posthog-code/download-posthog-code)仍是 waitlist/coming soon，不能写成普遍可下载成品 |

**开箱程度与费用：** Cloud UI、基础 PostHog AI、Hosted MCP、CLI 和受支持 AI wrapper 相对易用；Self-driving 需 GitHub、数据源、AI data processing，Slack 需管理员授权，Judge/Playground 完整模型需 BYOK。Self-driving、AI credits、Enterprise MCP 治理等是 Cloud 计费/专有服务，不能因主仓源码可见就推断 Hobby self-host 同样可用。

**同类差异：** 相比 Amplitude Agent Analytics，PostHog 更鲜明地把产品信号推到人工门禁 PR 再回测；相比 Langfuse，优势是 Trace 可直接关联产品行为、收入、Replay、Flag 与实验，但离线 Dataset、CI regression、annotation queue 等专用 LLMOps 面更薄；相比通用 Coding Agent，其独特价值是主动发现工作和生产上下文，而非更强的代码生成。

**易用性与限制：** Web UI、MCP、CLI、REST/API、跨 Web/移动/后端 SDK、Provider wrappers、模板、recipes 和 Demo 很丰富。Self-driving 会误判、只擅长维护/小优化且必须人工合并；AI 原始 input/output 只保留 30 天；Privacy Mode 会同时牺牲诊断上下文；Prompt 管理仍 Beta；Hobby self-host README 约 100k events/月的保守口径与其他决策图不一致，且没有支持/SLA。

**为何今天关注：** 事实是 README 已把定位升级为“building self-driving products”，MCP、AI Observability 和 human-gated loop 都有可核验文档，近期又高频发布 `agent-skills`。[Releases](https://github.com/PostHog/posthog/releases) 推断上，“产品数据平台 + Agent 执行闭环”比又一个 Coding Agent 更有差异；真正值得试的是 MCP、业务结果关联和人工门禁闭环，而不是“无需人类”的口号。

## 4. UI Skills

**一句话定位：** 面向使用 Coding Agent 做前端/UI 的设计工程师，把多个仓库的 `SKILL.md` 整理成可搜索、可按主题筛选、可预览的垂直目录，并用“先分类、再只加载 1–3 个 Skill”的路由提示词减少无关上下文；它不是 UI 生成模型或自动修复器。[README](https://github.com/ibelick/ui-skills#readme) · [官网](https://www.ui-skills.com/)

**典型场景：** 将模糊 UI 需求路由到 motion/accessibility/visual/framework；比较多个 Skill 再安装；用 `baseline-ui` 清理 AI 生成界面；专项修复无障碍、元数据和动画性能；给团队 Agent 固化 UI 规则。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **`start` 路由 Skill** | 打印 `ui-skills-root`，要求 Agent 先判断任务、选主题、优先 1 个且最多 3 个 Skill → 降低上下文污染 → 模糊/跨域 UI 任务 | **开箱即用**；只是提示词，不自动启动、分类、安装或执行。[路由协议](https://github.com/ibelick/ui-skills/blob/main/skills/ui-skills-root/SKILL.md) |
| **CLI 分类与清单** | `categories/list/list --category` 列主题与候选 → 终端内快速缩小范围 → Agent 自助发现 | `v0.2.3` 只有四个只读命令；需 Node/npm。[CLI](https://github.com/ibelick/ui-skills/blob/main/bin/ui-skills.ts) |
| **`get` 与重名消歧** | 输出完整 Markdown，重名时用 `author/slug` → 安装前审阅或临时喂给 Agent → 安全检查/自动化 | 本地 Skill 可读；第三方需联网且 raw URL 有效；打印不等于启用 |
| **Web 目录与全文搜索** | 官网按主题、作者和正文搜索，并支持键盘导航 → 更适合按问题发现 UI 指令 → 人工选型 | 官网本次显示 181 项；[Skills](https://www.ui-skills.com/skills) |
| **详情预览与来源追溯** | 展示用途、完整正文、作者、原始 GitHub、复制和相关 Skill → 先看规则再装 → 第三方指令审查 | 已可用；上游通常指向可变分支，不是版本冻结快照 |
| **一行安装到 Agent** | 详情页生成 `npx skills add <repo> --skill <name>` → 从发现过渡到持久复用 → 团队/项目规范 | 安装能力属于外部 [Vercel `skills` CLI](https://github.com/vercel-labs/skills#readme)，不是 `ui-skills` 自身 |
| **20 个宿主入口** | 官网为 Codex、Claude Code、Cursor、Copilot、Gemini 等给用途页 → 降低迁移心智 → 多 Agent 团队 | 是兼容说明，不是 20 个原生插件或逐项验收。[Agents](https://www.ui-skills.com/agents) |
| **机器可读入口** | `registry.txt`、`llms.txt` 和单 Skill 文本 → 脚本/Agent 无需解析页面即可摄取 → 自建检索/同步 | 已提供；没有正式 JSON API、鉴权 API 或 SDK。[Registry](https://www.ui-skills.com/skills/registry.txt) |
| **`baseline-ui`** | 对组件、交互、动效、排版、布局、性能给强规则和代码级修复格式 → 快速做一致清理 → React/Tailwind 页面 | 可直接用，但有明显 Tailwind/motion/Base UI 等栈与审美偏好。[原文](https://github.com/ibelick/ui-skills/blob/main/skills/baseline-ui/SKILL.md) |
| **窄域修复 Skills** | Accessibility、Metadata、Motion Performance 分别聚焦命名/键盘、SEO/OG、布局抖动/合成层 → 输出更聚焦 → 上线前检查 | 是检查清单，不是 axe/Lighthouse、WCAG 认证或性能 Profiler |
| **`improve-ui` 证据门控审计** | 只在设计契约、运行时关联和唯一确定修正都成立时报告最多 3 项，并产出计划 → 抑制凭审美大改 → 现有产品 Audit | main/官网可读；`npx ui-skills@0.2.3 get improve-ui` 实测失败，npm 尚未同步。[Skill](https://github.com/ibelick/ui-skills/blob/main/skills/improve-ui/SKILL.md) |

**开箱程度：** 官网和 `start/categories/list/get` 无需 API Key；持久安装需外部 `skills` CLI、写权限和宿主正确加载；第三方 Skill 可能要求具体框架、浏览器或付费模型。评分、统一自动 QA、before/after、完整 WCAG 2.2、部分宿主适配仍是开放 Issue/PR，不能写成现有能力。

**同类差异：** 相比 `skills.sh` / `vercel-labs/skills`，UI Skills 强在 UI 垂直策展、正文搜索、完整预览和最小上下文路由；弱在自己不能 add/use/find/remove/update，也无版本锁、评分、JSON 输出和生命周期管理。两者实际互补。

**易用性与限制：** Web UI 强，CLI 窄，无正式 API/SDK；README 很短但官网预览完整。官网 181 项/26 主题与 npm `0.2.3` 实测 110 项/24 主题不同步；第三方内容来自可变 raw URL，无评分和统一安全/质量门禁；Skill 只是指导，宿主工具、权限和依赖不足时不会自动获得能力。

**为何今天关注：** 事实是目录快速扩张，近期 PR 聚焦 WCAG、Registry 和 before/after，`v0.2.3` 修复 Windows `start`。[Release](https://github.com/ibelick/ui-skills/releases/tag/v0.2.3) 推断上，“AI UI 千篇一律”的痛点与 Skill 生态爆发共同推动传播。今天最值得体验的是 `start → category → 1 个窄 Skill → 审阅原文 → 再安装`，而不是全量安装。

## 5. AI Engineering From Scratch

**典型场景：** 走完整 AI Engineering 学习路线；按能力定向补 Agent/MCP；运行小型确定性 Demo；用 Capstone 建作品集/训练营；把 Skills/Workbench 带进现有 Coding Agent。

**为何今天关注：** 事实是项目今天 +191，公开 503 Lessons、388 Skills、99 Prompts、17 Capstone，Agent/MCP/Computer Use/安全热点集中；但主分支最近提交为 6 月 25 日、无当天 Release。推断上，大而清晰的免费/MIT 价值主张和“课程→可安装 Skill/Workbench”差异推动再传播，未必是新 Feature 事件。

## 6. Code Review Graph

**一句话定位：** 面向使用 AI 编码助手维护中大型、多语言仓库的开发者，以本地持久代码关系图向 Agent 提供审查、影响面、测试和多跳依赖上下文，减少重复整库阅读；它首先是代码理解/审查工具，不是通用自动修复 Agent。[README](https://github.com/tirth8205/code-review-graph#code-review-graph)

**典型场景：** PR/本地 Diff 预审；重构前 Blast Radius；新人快速理解关键流；本地优先代码检索；在 GitHub Action 中按风险提示/门禁。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **最小审查上下文** | `get_minimal_context/review_context/impact_radius` 从 Diff 返回相关函数、调用方、依赖、测试并限 token → 先缩小再精读 → PR/重构 | 核心可用；质量取决于语言关系和图新鲜度。[Commands](https://github.com/tirth8205/code-review-graph/blob/main/docs/COMMANDS.md) |

| **多跳结构查询** | 查询 callers/callees/importers/inheritors/tests，BFS/DFS、hub/bridge/耦合 → 一次回答多跳问题 → 故障定位/复杂重构 | 动态派发、元编程、重载不是编译器精度，并标置信度 |

| **广泛语言/Notebook/配置** | 覆盖 Python、JS/TS、Go、Rust、Java、C/C++、C#、SQL、Terraform、Vue/Svelte、Jupyter 等 → 一仓处理多技术栈 → Monorepo | 支持深度不一；自定义语法缺跨文件/框架语义 |
| **30 MCP Tools/多客户端** | stdio/HTTP MCP、5 prompts、3 workflow，自动写多种客户端配置 → 多 Agent 复用同一图 → Codex/Claude/Cursor 等 | 已实现但平台可靠性不齐；Windows/Copilot/Windsurf/VS Code 有开放兼容报告 |
| **可视化/导出/Wiki** | D3 图、JSON/GraphML/SVG/Obsidian/Neo4j、VS Code 树与 Blast Radius、社区 Wiki → 人类可核查/分享 → 架构导览 | CLI/HTML 可用；VS Code 需 backend，LLM Wiki 需 Ollama |
| **GitHub Action** | Runner 上构图并给 PR sticky comment，可按 high/critical 失败 → 无第三方 SaaS 也有一致风险提示 → CI | `v2.3.6+` 已发；不做 LLM 缺陷审查或自动修复。[Action](https://github.com/tirth8205/code-review-graph/blob/main/docs/GITHUB_ACTION.md) |
| **受控 Rename** | `refactor_tool` 预览 rename/dead code/suggest，`apply_refactor_tool` 用 `refactor_id` Dry-run/写入 rename → 先看 Diff 再改 → 小范围重命名 | 确有写入，但仅核实受控 rename；不是通用修 Bug/自动应用 Review |
| **多仓注册/跨仓搜索** | 注册多个 Repo/Alias 并跨仓检索，Daemon 同时维护 → 适合微服务/多 Worktree → 分散仓库 | 可用；每个 Worktree 应独立图，当前跨仓排序有 Issue |

**Benchmark 校正：** README 的约 **82× 中位数**是“整个源码 token ÷ 5 个搜索命中+邻边响应”，证明相对整库全读很省，不等于相对成熟 Agent 的 grep/top-file 阅读固定省 82×；正式小 Diff Benchmark 甚至可能 `<1`，即图响应比直接读文件更费 token。Impact recall 1.0 的 Ground Truth 又由同一图边派生，不能解读为真实影响面不漏。[复现说明](https://github.com/tirth8205/code-review-graph/blob/main/docs/REPRODUCING.md)

**开箱程度：** Python 3.10+ 安装后，本地 SQLite、Build/Update/Search/Impact、FTS、HTML、导出和 MCP 不需付费服务。Embedding、VS Code、LLM Wiki、GitHub Action 各有额外依赖；核心无商业付费层。GitHub App/org bot、团队共享图、>50k 文件专项优化仍在 [Roadmap](https://github.com/tirth8205/code-review-graph/blob/main/docs/ROADMAP.md)。

**同类差异：** LSP/Serena 的单符号与安全 Rename 更精确；向量 RAG 更适合概念/文档；Repomix 和 grep 对小仓/一次性问题更简单。CRG 的独有组合是本地持久跨语言结构图 + Review 导向影响/风险/测试/流 + MCP/CLI/CI。

**易用性与限制：** CLI 最完整，另有 D3/VS Code、MCP、模板、详尽文档和固定 SHA Benchmark。官方披露 keyword MRR 0.35、flow recall 33%、impact precision 约 0.578；大仓初建可能很慢；图会陈旧；小改不划算；云 AI 客户端调用 MCP 时返回内容仍可能离开本机。“local-first”不等于端到端不外发。

**为何今天关注：** 事实是 `v2.3.7` 刚在 7 月 18 日发布，加入 CLI-first、更多语言、传递测试覆盖、Churn/加权 Impact、Provenance 和安全卸载，并主动把 528×最大值下调为 82×中位数口径。[Release](https://github.com/tirth8205/code-review-graph/releases/tag/v2.3.7) 推断上，本地优先、MCP、多客户端和大仓 token 成本的组合击中当前 Coding Agent 痛点。

## 7. G0DM0D3

**一句话定位：** 面向 LLM 红队、安全研究者和重度模型玩家的开源多模型聊天界面，用并行竞速、越狱提示组合、输入扰动和二次改写寻找更直接的回答；它不是带知识库、Web 搜索、工具/MCP 和工作流的通用 Agent 平台。[README](https://github.com/elder-plinius/G0DM0D3#readme)

**典型场景：** 同题多模型横评与选优；授权范围内测试拒答/过滤鲁棒性；用本地模型做隐私实验；比较采样/提示策略；通过 OpenAI-compatible API 把竞速接到现有客户端。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **多 Provider 入口** | 同页配置 OpenRouter、Venice 和 localhost OpenAI-compatible 服务 → 比较云端/本地模型 → 横评/红队 | 已可用；不是各家原生 API 直连，需 Key 或本地服务。[本地指南](https://github.com/elder-plinius/G0DM0D3/blob/main/LOCAL_MODELS.md) |
| **ULTRAPLINIAN 竞速** | FAST→ULTRA 并发查 12–60 个 OpenRouter 模型，并可追加 Venice/本地模型，用自定义 100 分选胜者 → 用调用量换多样性和初筛 → 复杂问题/选型 | 已有、实验性；越大越慢、越贵、越易限流 |
| **Classic 组合竞速** | 将预设拒答逆转提示与模型并行 → 提供可复用红队基线 → 安全回归 | 已有；README/UI 对 4/5 组有漂移，“battle-tested”不是成功率保证 |
| **Parseltongue** | 用 leetspeak、Unicode、盲文、摩斯等 33 种输入扰动，分三级并可拒答重试 → 系统测试表面混淆鲁棒性 → 内容过滤/注入研究 | 研究用途；不保证绕过或正确，调用数与成本上升 |
| **AutoTune** | 将问题归入 20 类并自动选 temperature/top-p/top-k/penalty，也可手调 → 减少新手调参 → 代码/创作/分析 | 项目启发式，无公开最优基准；Provider 可能忽略参数 |
| **Liquid/Coaching/候选浏览** | 先显示当前最佳，达到阈值的新候选实时替换，并保留替代回答和历史版本 → 改善等待感且能人工复核 → 长回答打磨 | UI 已有，按 Research Preview；额外 Judge/Coach 增加费用；**不是聊天分支** |
| **本地模型/Local-only** | 发现 `/v1/models` 并调用 `/v1/chat/completions`，最多 8 个本地模型；Local-only 排除云 Provider、关闭应用遥测并本地评判 → 缩小数据外发 → 私密实验 | 需 Ollama/LM Studio/llama.cpp/vLLM 等、硬件和 CORS；首个本地模型承担 Judge |
| **本地历史/备份/No-Log** | 对话与设置存 `localStorage`，JSON 导入导出、清空、停应用遥测 → 无账号自管记录 → 个人实验 | 已有；约 5MB、100 会话/每会话 500 消息上限；无云恢复，Key 不是安全密钥库 |
| **图片输入/Vision 摘要** | 先用 Gemini Flash 或本地 Vision 模型生成图片描述，再注入竞速上下文 → 让不支持图像的候选也处理截图 → 视觉问答 | 已存在但文档弱；摘要会丢细节，不等于每个模型原生看图 |
| **单文件 UI + 可选 API** | `index.html` 即可静态托管；Node/Express API 提供 OpenAI-compatible Chat Completions 和竞速模式 → 低门槛自托管/SDK 接入 | UI 易用；API 是 Research Preview，无专用 CLI/SDK。[API](https://github.com/elder-plinius/G0DM0D3/blob/main/API.md) |

**不存在的常见能力：** 当前未提供聊天树分支、Artifacts、联网/会话搜索、文件知识库、RAG、MCP、工具注册、浏览器/代码执行或 Agent Action。竞速替代回答、代码块渲染和“research tool”产品类别都不能被写成这些 Feature。

**开箱程度与费用：** 托管 UI、本地历史、备份和 No-Log 可直接看；真正调用至少需 OpenRouter/Venice Key 或本地模型服务。源码不收费，但一次 ULTRAPLINIAN/Parseltongue/Judge/Coach/Liquid 会放大模型账单；本地模型则付硬件、电力与维护成本。项目无可核验正式商业套餐。

**同类差异：** 相比 LibreChat/Open WebUI，G0DM0D3 强在几十模型竞速、低拒答红队、33 种扰动和单文件部署；明显缺 Provider 广度、用户/团队、会话搜索、RAG、Web Search、MCP、Artifacts、工具与代码执行。适合红队实验，不适合日常 Agent 工作台。

**易用性与限制：** 有托管 Web UI、OpenAI-compatible API、预设和在线 Demo；无 CLI/自有 SDK/Release/Changelog/独立 Benchmark。自动评分偏好少拒答和少 hedging，可能奖励自信但错误/不安全回答；应用遥测默认开启，No-Log/Local-only 也不能关闭 Provider/托管基础设施日志；浏览器 Key/历史有同源脚本和扩展风险；AGPL 网络部署需评估源码义务。

**为何今天关注：** 事实是项目 3 月才创建，本周合入 Venice-only、本地模型与更明确隐私说明。[PR #65](https://github.com/elder-plinius/G0DM0D3/pull/65) · [PR #69](https://github.com/elder-plinius/G0DM0D3/pull/69) 推断上，“uncensored/liberated AI”、几十模型赛跑、33 种扰动和 Local-only 极易截图传播；这解释热度，不证明安全、正确或生产成熟。

## 8. AirLLM

**一句话定位：** 面向想在低显存消费设备上偶尔运行远大于显存的 Hugging Face 因果模型的开发者，用极低峰值 VRAM 换巨大下载/磁盘占用和很慢生成；更像“让模型跑起来”的研究与离线工具，而非实时聊天或高并发服务引擎。[README](https://github.com/lyogavin/airllm)

**典型场景：** 4–12GB VRAM 上验证 70B/MoE/FP8 能否完成生成；允许分钟级等待的离线抽取/评分；对 checkpoint 做少量 sanity check；Apple Silicon/CPU 隐私实验；教学演示参数量与峰值显存的取舍。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **超低峰值 VRAM** | 项目宣称 70B 全精度约 4GB、405B 约 8GB、671B 约 12GB → 无需一次装下权重 → 极端低显存试跑 | 核心主张；**未给绑定硬件/Revision/耗时的可复现官方结果** |
| **统一 `AutoModel`** | HF Repo ID/本地目录用 `from_pretrained()/generate()` → 接近 Transformers 心智 → 脚本迁移 | PyPI `3.0.1` 可用 |
| **广模型族** | Llama 2/3/4、Qwen 1/2/3、DeepSeek V2/V3/R1、Mistral/Mixtral、Phi、Gemma 等 → 一入口覆盖主流 Causal LM → 兼容试验 | `v3.0` 已扩展；不等于任意 checkpoint/发布当天必可用 |
| **现代 Transformers 生成** | 更新 Transformers/Accelerate，并支持常见 `generate()` 参数 → 降低旧脚本迁移 → Python 推理 | `v3.0` 已重做；版本有明确约束 |
| **预量化 FP8** | 读取 DeepSeek-V3/Qwen3-FP8 等原生 FP8 checkpoint → 降单层和磁盘负担 → 新式超大 MoE | 已可读预量化权重；**不是通用 FP8 转换器** |
| **可选 4/8-bit 压缩** | 准备分层文件时压缩 → 减少存储与搬运 → 磁盘紧张试验 | 需 bitsandbytes，通常走 CUDA；与 Prefetch 不兼容；“3×/几乎无损”证据不足 |
| **自动下载/Gated Token** | 直接拉 HF checkpoint，`hf_token` 访问已获批模型 → 少手工路径操作 → Llama 等 gated Repo | 需网络、许可、带宽和足量磁盘 |
| **分层保存/删原件** | 指定大盘保存转换结果，可选删原始权重 → 管理数百 GB 占用 → 超大模型 | 路径可配；`delete_original=True` 有“双删后崩溃”开放缺陷，需备份小测 |
| **非分片 Checkpoint** | 支持单文件 PyTorch/Safetensors → 小模型/自定义导出更易接 → 兼容验证 | 已发布；未承诺 GGUF/AWQ/GPTQ 任意格式 |
| **Prefetch/Profiling** | 尝试重叠读取和计算，输出耗时 → 定位磁盘/传输/计算瓶颈 → SSD/RAM Disk 选型 | 已有；旧文档只称约 10%，不是通用吞吐加速 |
| **CPU/Apple Silicon** | CPU 路径与 Apple Silicon MLX → 无 NVIDIA 也能低频本地跑 → 隐私实验 | 文档/兼容弱；Quickstart 仍 `.cuda()`，Intel Mac 不支持，AMD 有落 CPU 报告 |
| **Notebook/测试 Harness** | 通用、macOS、405B Notebook 与可设 VRAM cap/计时脚本 → 自己收集峰值和耗时 → 首次试跑 | 示例已有；405B Notebook 无执行输出，不能当性能证据 |

**最重要的成本边界：** “70B 单 4GB GPU”只是峰值 VRAM，不是 4GB 电脑。70B BF16/FP16 原始权重粗算约 140GB，首次还要保存逐层副本；README 自己警告转换“very disk-consuming”。维护者确认吞吐受逐层磁盘读取限制，压缩主要省内存/磁盘，不意味着高 tokens/s。[维护者说明](https://github.com/lyogavin/airllm/issues/117#issuecomment-4848716309)

**405B/FP8 校正：** 官方 405B Notebook 实际加载 `bnb-4bit` checkpoint，且没有运行输出、显存截图、硬件、耗时或 revision，不能证明 405B 全精度 8GB。[Notebook](https://github.com/lyogavin/airllm/blob/main/air_llm/examples/run_llama3.1_405B.ipynb) `v3` FP8 是消费预量化 checkpoint，不是把任意模型转 FP8。

**同类差异：** 相比 Transformers/Accelerate 常规 Offload，AirLLM 峰值显存更低但性能取舍更极端；相比 llama.cpp/GGUF，可保留 HF BF16/FP16 或 FP8，但缺成熟 CLI/Server/UI 和交互速度；相比 Ollama/LM Studio，无模型管理/聊天/API；相比 vLLM，目标相反，没有批处理、并发、调度、分布式和低延迟。

**易用性与限制：** 有 PyPI、Python API、HF 集成、Notebook、FAQ 和测试脚本；无 UI、正式 CLI、HTTP/OpenAI Server、Docker 部署、Ollama/OpenWebUI、多用户调度和可信当前性能表。若模型可装入 RAM/VRAM，AirLLM 通常不划算；建议从 7B/32B 测完整转换/耗时后再下载 70B+。

**为何今天关注：** 事实是 `v3.0/3.0.1` 最近补入 Qwen3、DeepSeek V3、FP8/MoE 和现代 Transformers，并展示 235B≈3GB、671B≈12GB 等高传播数字。[v3.0](https://github.com/lyogavin/airllm/releases/tag/v3.0.0) 推断上，低显存跑超大模型的反差与 GPU 成本痛点推动收藏；热度反映“能不能跑”的想象力，不代表生产吞吐。

## 9. wigolo

**一句话定位：** 给 Coding Agent 与自托管 Agent 使用的本地优先 Web 工具层，在用户机器上提供搜索、网页读取、爬取、结构化提取、研究和变更监控，并通过 MCP、CLI、REST、SDK 暴露；它不是离线全网搜索引擎或大规模托管爬虫。[README](https://github.com/KnockOutEZ/wigolo) · [Tools](https://github.com/KnockOutEZ/wigolo/blob/main/docs/tools.md)

**典型场景：** Coding Agent 查最新官方资料；把文档站 Crawl 为本地可复用 Cache；把 JS 页面/PDF 转成 Markdown/结构化 JSON；生成有来源的 Research Brief；Watch 版本/价格/文档变化。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **多源 `search`** | General/News/Code/Docs/Papers/Images，多查询、域名/时间/精确短语过滤，返回引用、评分、引擎状态与新鲜度 → Agent 可判断证据和退化 → 最新技术调研 | 默认公开源无 Key；`format=answer` 高质量合成需 LLM。18 adapters 是跨垂类总数 |
| **分层 `fetch`** | URL→干净 Markdown，不足时升级浏览器；支持 JS/PDF/Section/Screenshot/Actions/已有会话 → 一工具覆盖常见页面 → 文档、SPA、登录页 | 公开网页基础可用；首次下载浏览器；认证需 CDP/Profile/Storage State，顽固反爬无保证 |
| **`crawl` / `map`** | BFS/DFS/Sitemap/只列 URL，限制深度、页数、正则并写本地 Cache → 快速构建文档语料 → RAG/站点盘点 | 默认约 20 页，REST 最大 200 页/深度 5；Research-grade，不是规模采集 |
| **无 LLM `extract`** | 抽表格、KV、JSON-LD、Metadata、Selector、品牌素材，内置多类 Schema 或自定义 JSON Schema → 常见提取更确定且免模型费 → 商品/论文/文章入库 | 结构模式可用；缺标记时覆盖有限；LLM fallback 需本地/云模型 |
| **本地 `cache/find_similar`** | 持久保存页面，FTS/语义混合检索、统计、清理、变化检查和相似页 → Web 记忆跨会话、缓存命中可离线 → 重复技术域/RAG | 已可用；语义需本地模型和语料，发现新页仍联网 |
| **`research`** | 拆子查询、并行搜索、交叉核验，输出 Topics/Findings/Comparison/Gaps 和最多 50 来源 → 更系统地交付证据包 → 选型/政策追踪 | 无 Key 主要给 structured brief/evidence；Essay-grade 合成需云 LLM 或 Ollama |
| **有预算 `agent`** | 在页数/时限/Schema 内计划搜索、抓取、提取并返回步骤/来源/警告 → 少手工编排 → 字段收集/研究包 | Gather loop 可无 Key；最长任务被 clamp 到 240 秒，不是长期后台 Agent |
| **`diff/watch`** | 比缓存、两个 URL 或文本；Watch 定时重抓并内联或 Webhook → 低成本跟踪页面变化 → Changelog/价格监控 | Diff 可用；Watch 是惰性调度，必须保持 Serve/MCP 进程，不是独立 Scheduler |
| **MCP/CLI/REST/SDK** | 10 工具通过 stdio/HTTP/SSE、CLI/NDJSON、OpenAPI、TS/Python SDK，另有 LangChain/CrewAI/LlamaIndex/Vercel 封装 → 一份 Cache 多宿主复用 → Agent/自动化 | `0.2.0` 已发；REST Research/Agent 的 `stream` 不实时，只返回整包 |
| **初始化/诊断/Skills** | `init --agents` 下载组件、写配置，`doctor/verify --fix`，TUI Config 和 11 工具 Skills → 压缩首次配置 → 本地部署/CI | Node 20+、约 1.5GB 磁盘；无 Web GUI，“dashboard”是终端配置别名 |

**“Local-first / No Key / $0”校正：** 程序、Cache、向量、配置默认在 `~/.wigolo/`，无 wigolo 后端/账号/计费；但它**不是离线**：搜索词发给 Bing/DuckDuckGo/Wikipedia/GitHub/arXiv 等公开源，目标请求发给网站，初始化下载浏览器/模型，云 LLM 会接收合成内容。无 Key 仅表示基础搜索、公开抓取、启发式提取、Cache、Diff 等不需要商业 Key；Brave、更高 GitHub 限额、云 LLM、登录会话各有凭据。$0 不含硬件、电力、网络、代理和付费 LLM。[隐私](https://github.com/KnockOutEZ/wigolo/blob/main/docs/privacy-security.md)

**索引边界：** wigolo 没有自建全网索引；“本地索引”只覆盖这台机器已经 search/fetch/crawl 的页面。公开搜索源会限流、改版和阻断，无 SLA；复杂反爬/IP Reputation 可能仍失败，尤其 VPS。

**同类差异：** 相比 Firecrawl，wigolo 无账号/Key并有本地 Cache，但缺托管浏览器、代理网络、SLA、Playground 和大规模吞吐；相比 Tavily，无按调用计费、本地 Memory/Diff/Watch 更强，但托管稳定性和无 Key 合成更弱；相比 Exa，缺 Websets 式实体/enrichment/UI/团队产品，胜在个人 Agent 的本地 Web 层。

**易用性与限制：** 有 CLI/REPL/TUI、MCP、REST/OpenAPI、SDK、框架封装、Skills、10 组示例和录屏；无 Web 搜索 UI/团队控制台/托管服务。`v0.2.0` 是 Public Beta；GitHub Release 无 Binary Assets，Homebrew/单文件/MCPB/托管脚本仍未来；Chrome Profile 显式路径有复制未清理 Issue；OpenCode 自动配置未核实；README 单次 Benchmark 不是独立多数据集评测。

**为何今天关注：** 事实是 `v0.2.0` 7 月 17 日发布，集中增加零配置 Onboarding、REST/OpenAPI、SDK/框架、Skills、CLI Shell 和安全加固，18 日又补完整文档与无 LLM 降级。[Release](https://github.com/KnockOutEZ/wigolo/releases/tag/v0.2.0) 推断上，“No API Keys / $0 / Local-first + 一键接热门 Coding Agent”非常易传播，也击中 Agent 循环放大搜索账单的现实痛点。

## 10. Kimi CLI

**一句话定位：** 面向希望在终端、浏览器或 ACP IDE 中让 AI 直接读写代码、执行命令并验证的开发者，把 Coding Agent、会话/子代理、MCP/Skills 与 Headless 自动化放进本地 CLI；但官方已宣布本仓库逐步停止并演进为 [MoonshotAI/kimi-code](https://github.com/MoonshotAI/kimi-code)，新选型应优先评估后继项目。[README / 迁移公告](https://github.com/MoonshotAI/kimi-cli/blob/main/README.md)

**典型场景：** 功能开发与重构；只读探索/Plan 后修 Bug；多子代理/后台任务并行；用 JSONL/Wire 嵌入 CI 与上层工具；在终端、Web、Toad TUI 和 ACP IDE 之间恢复会话。

**开箱程度：** 安装并选择 Kimi OAuth 或其他 Provider Key 后，终端编码闭环、审批、Plan、会话、子代理、后台任务、`/init`、导出和 Web UI 可用。MCP、IDE、Skills/Flows、LAN Web 各需配置。`okabe`、Wire、Plugin/Hooks、Visualizer 明确非稳定。

**生命周期是最大边界：** 最新稳定版是 [1.49.0](https://github.com/MoonshotAI/kimi-cli/releases/tag/1.49.0)（7 月 16 日），修复长回合 Completion Budget/Thinking 历史；但旧仓库正在 wind down，新安装会迁移配置和会话。今天首次安装不应只凭 Stars 押注旧仓，应比较后继 `kimi-code` 的 Feature/迁移成熟度。

**同类差异：** 相比 Claude Code/Codex/Gemini CLI，Kimi CLI 的强项是 Terminal + Toad + 本地 Web + ACP + Trace Viewer，多 Provider、持久子代理、Flow、Hooks/Wire 也很宽；缺口是没有核实到内建沙箱/细粒度命令 Policy，协议和扩展多处非稳定，且迁代带来长期风险。

**易用性与限制：** 英中双语文档、CLI/TUI/Web、JSONL/Wire/ACP、示例和 Release Notes 完整。审批不是沙箱；Print/AFK 会直接执行；MCP 有供应链/Prompt Injection；Skills 是提示而非确定路由；非 Kimi Provider 的搜索/Thinking/多模态/费用统计不等价；Python 3.12–3.14，Toad 3.14+，Windows 需 Git Bash；匿名 Telemetry 默认开启但可关。

**为何今天关注：** 事实是 1.49.0 三天前发布，Feature 面一次覆盖子代理、Skills、MCP、ACP、Headless、Web UI 与可观测；同时官方公布迁代。推断上，新版与迁移公告共同带来回看流量，而不代表旧仓库恢复长期高速开发。今天关注它的价值更像研究一套成熟 Terminal Agent 如何收敛迁代，而不是直接做长期新选型。

## 今天最值得体验的 3 个项目

### 1. Code Review Graph：最适合马上在真实仓库验证

- **最吸引人的 Feature：** `detect-changes/get_review_context/get_impact_radius` 把 Diff、Callers、Tests、Flows 和 Risk 组合成可审上下文；本地持久图可被多个 MCP 客户端与 GitHub Action 复用。
- **为什么入选：** 不需要新的模型平台或付费 SaaS，功能与限制都有可复现材料；它解决的是 Coding Agent 在大仓反复搜索的真实成本。
- **建议体验法：** 选一个 500–3,000 文件、真实多文件 Diff 的仓库，先跑 `build`，再用 `--brief --verify` 对比人工 grep。重点看召回的 Callers/Tests/Flows 是否可信，**不要只看 82×数字**。

### 2. wigolo：最有产品感的本地 Agent Web 层

- **最吸引人的 Feature：** `search → fetch/crawl → cache → find_similar/diff` 把公开 Web 证据沉淀为本地可复用记忆，并从 MCP/CLI/REST/SDK 同时暴露。
- **为什么入选：** 无需先买商业搜索 Key，10 个工具和完整例子能很快进入 Coding Agent；“证据位置、引擎退化、缓存新鲜度”比只返回一个答案更适合审计。
- **建议体验法：** 先测常用官方文档站的 Search/Fetch/Crawl、中文与垂类质量，再分别比较无 LLM、免费 Gemini、本地 Ollama 的 Research 输出；按 Public Beta 管理，不把“Local-first”理解成离线。

### 3. LingBot-Map：今天最有视觉冲击的新能力

- **最吸引人的 Feature：** 长视频 Streaming + Keyframe/Windowed 降级 + 一体化点云 MP4/YAML 镜头，使 3D 重建从 Benchmark 走到可展示产物。
- **为什么入选：** 榜首项目不仅有论文，还给出权重、四个小例子、25k 帧实例、渲染和 Benchmark，适合快速判断空间智能模型对自己的数据是否有价值。
- **建议体验法：** 先跑仓内小例子和 Viewer，再用 1–3 分钟自有视频测几何/显存；只有通过后再投入长视频与重渲染依赖。不要把它当完整 SLAM、NeRF/3DGS 前端或成品 3D 资产管线。

## 值得持续跟踪的新 Feature 方向

1. **持久本地 Context 的生命周期。** Code Review Graph 的代码图和 wigolo 的 Web Cache 都在把“每轮重搜”变成跨会话资产；下一步关键是新鲜度、版本/Worktree 隔离、可删除性、加密与可证明的数据外发边界。
2. **Agent Grounding 从 Schema 升级为业务语义合同。** Ossie 的 `ai_context`、指标、关系和同义词说明，仅有表结构远远不够；值得跟踪 verified queries、权限、认证指标、Conformance Test 和查询 Runtime 是否真正落地。
3. **Human-gated Agent 闭环的结果验证。** PostHog 把 Signal→Report→PR→合并后指标验证串起来；比“Agent 会改代码”更重要的方向是误报率、PR 可合并率、单位成本、回滚和真实业务结果。
4. **Skill Registry 的质量供应链。** UI Skills 暴露了目录与 CLI 同步、第三方可变 URL、评分、版本锁、宿主兼容和效果评测缺口。未来 Skill 生态需要签名、冻结版本、自动安全审计和可复现 before/after，而不只是更多条目。
5. **MCP + ACP + Headless JSONL 的协议组合。** 今天多个项目都同时提供 MCP、CLI、REST/SDK 或 ACP；真正值得跟踪的是审批语义、Prompt Injection 防护、能力协商、同一会话跨 UI 恢复，以及不同客户端的兼容测试。
6. **低资源推理必须披露“总资源与延迟”。** AirLLM 说明 VRAM 数字不能替代原始权重、转换副本、磁盘带宽、首 Token/每 Token 延迟和并发。今后更可信的 Feature 页应给硬件、Revision、上下文、总磁盘、完整耗时与复现脚本。
7. **空间模型从短序列 Benchmark 走向流式长视频产物。** LingBot-Map 的 Keyframe/Window 和成片管线值得关注；下一阶段看回环/全局一致、动态 4D、行业格式、训练/微调和 ROS/3DGS 等下游接口。
8. **Agent Web 层从“搜索答案”走向“证据、缓存与变化”。** 搜索源状态、引用位置、Cache 新鲜度、Diff/Watch 和无 LLM 降级，可能比单次答案质量更适合长期 Agent；同时要透明说明搜索词、页面与合成内容分别发给谁。
