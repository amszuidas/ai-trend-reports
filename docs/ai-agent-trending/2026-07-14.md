## 项目 Feature 深度卡片

### 1. Vibe-Trading

**一句话定位：** 面向量化爱好者、交易研究者和需要自然语言金融分析的团队，把跨市场数据、策略回测、行为复盘和多 Agent 研究放在一个工作台中。

**典型场景：** 市场问答与报告；跨市场策略回测；Alpha 因子筛选；导入交易记录诊断行为偏差；通过 MCP、IM 或定时任务运行研究流程。

| Feature | 能做什么 | 用户价值与场景 |
|---|---|---|
| 自然语言金融研究 | 用问题启动行情、基本面、新闻和策略分析 | 降低脚本研究门槛；适合市场扫描和标的比较 |
| 19 类免费数据源 | 覆盖 A/HK/US/India 股票、加密、期货、外汇及本地文件，并提供回退 | 更容易启动跨市场研究 |
| 八市场策略回测 | 提供 Monte Carlo、Bootstrap、Walk-Forward、Run Card 及 Pine/TDX/MT5 导出 | 检查策略稳健性，而不只看单次收益 |
| Alpha Zoo | v0.1.11 提供约 460 个稳定因子 | 快速筛选和比较研究假设 |
| Research Autopilot | 持续生成、登记和验证研究假设 | 减少研究线索散落；调度默认关闭 |
| 多 Agent Presets | 提供约 30 种研究、风险、新闻、辩论和决策组合 | 快速尝试角色协作与观点交叉检查 |
| Trade Journal / Shadow Account | 导入券商 CSV，识别行为模式并生成反事实报告 | 复盘个人交易纪律；结论不是因果证明 |
| 多入口交互 | CLI、TUI、Web、REST、SSE、约 54 个 MCP 工具及多种 IM 适配器 | 嵌入桌面、服务器和聊天机器人环境 |
| 跨会话记忆与 Skills | 保存研究上下文和领域规则 | 长期跟踪同一策略或市场 |
| 券商连接与受控执行 | 通过 11 个连接器尝试打通研究到下单 | 缩短操作链路；并非所有连接器支持交易，实盘仍实验性 |

**开箱/依赖/实验：** CLI、Web、免费数据源、回测和本地文件可直接使用；自然语言能力需要云模型或 Ollama；部分数据、券商 OAuth、消息平台和 PDF 导出需额外配置；QVeris 可能付费。Options Lab、完整 Portfolio Studio 尚未全部完成，主分支的 Strategy Development Manager 与 Alpha 461 不能倒算进 v0.1.11。

**同类与缺口：** 相较 TradingAgents，它的入口和数据/回测覆盖更广；相较 OpenBB，更强调 Agent 工作流；相较 Qlib，更容易交互，但训练研究深度较弱。公共数据质量、数据源回退语义、因子可交易性和实验性实盘是主要边界。

**今天关注原因：** 事实是近期发布 v0.1.11，并持续修复研究工作流；推断是金融 Agent、低门槛回测和 Demo 传播性共同推动了 +1,153 日增。

来源：[README](https://github.com/HKUDS/Vibe-Trading)、[v0.1.11](https://github.com/HKUDS/Vibe-Trading/releases/tag/v0.1.11)、[Changelog](https://github.com/HKUDS/Vibe-Trading/blob/main/CHANGELOG.md)。

### 2. AIRI

**一句话定位：** 面向想拥有自托管数字角色、VTuber 助手或桌面伴侣的用户，把对话、语音、视觉、头像和工具使用放入一个跨平台容器。

**典型场景：** 桌面陪伴；语音角色聊天；直播/VTuber；屏幕视觉辅助；角色驱动的 Minecraft、Factorio 等游戏实验。

| Feature | 能做什么 | 用户价值与场景 |
|---|---|---|
| Live2D、VRM、Spine | 显示多种 2D/3D 角色格式 | 使用自有形象；适合 VTuber 和品牌角色 |
| 桌面伴侣模式 | 让角色在桌面窗口持续显示与交互 | 从网页聊天框进入日常桌面体验 |
| Character Card V3 | 标准化定义角色性格、背景和表达方式 | 角色更易迁移和共享 |
| 多模型与本地推理 | 选择云模型或本地模型 | 在效果、成本和隐私之间取舍 |
| STT/TTS 语音闭环 | 组合语音识别、模型回复与语音合成 | 适合陪伴、直播和无键盘使用 |
| 屏幕与窗口视觉 | 在授权且模型支持时理解屏幕内容 | 围绕用户正在看的内容交流 |
| MCP 工具与 Gamelets | 调用外部工具或参与特定游戏流程 | 从纯聊天延伸到行动 |
| 多平台客户端 | 覆盖 Web/PWA、桌面和移动方向 | 降低设备切换成本 |
| Artistry 图像生成 | 在角色互动中生成或编辑视觉内容 | 丰富角色表达和创作 |
| 平台与游戏连接 | 连接 Telegram、Discord、Minecraft 等环境 | 进入已有社区和游戏 |

**开箱/依赖/实验：** Web Demo、桌面客户端和基础角色可体验；云模型、STT/TTS、图像生成通常需要 API Key，本地推理需要硬件，屏幕视觉需要系统权限。Memory Alaya 与长期记忆仍在推进；Factorio 是 PoC，Helldivers 等仍为 WIP，KSP 属后续方向。

**同类与缺口：** 相较 Open-LLM-VTuber，AIRI 的角色格式、平台、MCP 和游戏方向更广，实时语音专注度较弱；相较 SillyTavern，更重桌面具身与工具，但角色扮演生态较弱。旧 Live2D 兼容、本地硬件、应用体积和提供商配置新鲜度是门槛；屏幕视觉不等于成熟 Computer Use。

**今天关注原因：** 事实是 v0.11.0 于 7 月 8 日发布；推断是可视角色、语音与游戏 Demo 具有强传播性，贴合具身 Agent 热度。

来源：[仓库](https://github.com/moeru-ai/airi)、[v0.11.0](https://github.com/moeru-ai/airi/releases/tag/v0.11.0)、[在线 Demo](https://airi.moeru.ai/)。

### 3. awesome-llm-apps

**一句话定位：** 面向学习者、原型开发者和需要快速验证 AI 产品想法的团队，提供可运行的 LLM、Agent、RAG、MCP 与多模态应用源码。

**典型场景：** 学习 Agent 框架；复制 SaaS 原型；验证 RAG；尝试 MCP；构建语音、多模态或常驻工作流。

| Feature | 能做什么 | 用户价值与场景 |
|---|---|---|
| 自包含模板 | 多数示例包含代码、依赖和运行说明 | 降低从概念到 Demo 的成本 |
| Starter Agents | 研究、旅行、金融、招聘等入门 Agent | 从完整任务而非空 SDK 开始 |
| Advanced Agents | 多角色、工具调用、自我改进和复杂任务拆解 | 理解 Agent 可交付的结果 |
| Always-on Agents | 定时运行、消息投递和持续监测 | 把一次性对话扩展为长期任务 |
| MCP 模板 | 演示连接外部服务和工具 | 降低 MCP 集成学习成本 |
| Generative UI | 让模型输出交互界面或同步共享状态 | 体验更接近真实产品 |
| 语音与多模态 | 组合语音、图像和视频理解 | 快速验证非文本体验 |
| RAG 变体 | 覆盖基础、图谱、纠错与 Agentic RAG | 按数据形态比较方案 |
| Memory / Chat-with-X | 会话记忆及文档、数据库、网站问答 | 快速形成定制知识助手 |
| Skills、微调与 SDK 教程 | 组合提示、数据准备和平台 SDK | 从原型继续扩展 |

**开箱/依赖/实验：** 源码与说明可直接获取；多数示例需要模型 API Key，部分需要 Ollama、Neo4j、GPU、Gmail OAuth、Playwright 或云调度。仓库没有统一 Release、配置、测试矩阵或生产部署保证。

**同类与缺口：** 比传统 Awesome List 更可运行，比单一厂商 Examples 覆盖更广，比低代码平台更透明；缺少统一 UI、运行时、权限和治理。整个仓库跨提供商不代表每个应用可随意换模型，“生产风格示例”也不等于生产级系统。

**今天关注原因：** 事实是首页突出 Agent Skills、常驻 Agent、语音和 Generative UI；推断是 +996 日增主要来自低试用成本，而非单一新版本。

来源：[README 与分类目录](https://github.com/Shubhamsaboo/awesome-llm-apps)。

### 4. Hallmark

**一句话定位：** 面向使用 Codex、Claude Code 或 Cursor 生成前端的开发者，用结构、审美约束和交付检查减少千篇一律的 AI 风格页面。

**典型场景：** 从 brief 生成页面；审计现有 UI；保留信息架构改版；从截图提取设计 DNA；设计带完整状态的单组件。

| Feature | 能做什么 | 用户价值与场景 |
|---|---|---|
| Brief 驱动 Build | 先理解受众、目的和语气再选择页面形态 | 减少模型直接套模板 |
| 21 种 Macrostructure | 从 Workbench、Manifesto、Long Document、Bento 等页面形状中选择 | 差异来自结构而非只换颜色 |
| 20+ 主题与 Custom | 使用目录主题或组合 OKLCH 配色和字体 | 兼顾速度和品牌适配 |
| 跨运行结构去重 | 读取 `.hallmark/log.json` 避免连续复用结构、导航和 Hero | 降低批量生成收敛成同一种网站 |
| Audit | 审查字体、颜色、结构、动效、响应式和 AI 痕迹 | 先评估再修改 |
| Redesign | 默认保留路由、文案意图、组件归属和信息架构 | 降低真实代码库改版风险 |
| Study | 从截图或公开 URL 提取结构、字体角色、色彩锚点和节奏 | 借鉴设计原则而非像素复制 |
| `design.md` | 将设计 DNA 固化为其他 Agent 可读取的系统 | 跨页面和工具复用设计决策 |
| 组件八状态 | 展示 Default、Hover、Focus、Active、Disabled、Loading、Error、Success | 减少状态遗漏 |
| Anti-slop 自检 | 检查层级、响应式、对比度、虚构指标等 | 把审美要求转成可检查规则 |
| Hero enrichment | 优先排版，必要时再用 CSS art、SVG、生成图或素材 | 避免装饰掩盖信息问题 |

**开箱/依赖/路线图：** Build、Audit、Redesign、Study、项目日志和组件八状态当前可用；必须依托兼容 Coding Agent，URL Study 依赖网页读取。图像 API Hook、Brand-first、Variant、多页一致性、Charts、Explain、MCP Live Preview 都仍在 Roadmap。

**同类与缺口：** 相较 Impeccable，Hallmark 更强调 Macrostructure、去重和 Study；Impeccable 的命令、扩展、确定性检测和发布体系更成熟。Hallmark 没有独立 API、SDK、浏览器编辑器或视觉反馈闭环。

**特别注意：** README/官网仍写 20 themes、57 gates，当前 Skill 已列出 22 主题，检查项计数又出现 58、65、66、69 等数字；因此保守表述为“20+ 主题、约 58 项检查”。

**今天关注原因：** 事实是当日新增 794 Stars，提供 MIT、跨 Agent 安装和可视 Demo；推断是 AI UI 同质化痛点与前后对比 Demo 推动传播。

来源：[README](https://github.com/Nutlope/hallmark)、[当前 Skill](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/SKILL.md)、[Macrostructures](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/references/macrostructures.md)、[Study](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/references/study.md)、[Roadmap](https://github.com/Nutlope/hallmark/blob/main/ROADMAP.md)。

### 5. Graphify

**一句话定位：** 面向 Coding Agent、开发团队和新接手代码库的工程师，把代码、文档、Schema、基础设施和媒体整理成可查询、可追溯、可共享的项目知识图谱。

**典型场景：** 陌生仓库 onboarding；追踪跨模块依赖；让 Coding Agent 先查图再读文件；团队共享上下文；评估 PR 影响和冲突。

| Feature | 能做什么 | 用户价值与场景 |
|---|---|---|
| 混合资料统一建图 | 处理大量代码语言及 SQL、Markdown、Office、PDF、图片、音视频 | 不必分别维护代码索引和文档 RAG |
| 本地确定性代码提取 | 代码 AST 可在本机处理 | 降低私有仓库的隐私和模型成本 |
| 关系证据标签 | 标明 `EXTRACTED`、`INFERRED`、`AMBIGUOUS` | 区分源码事实与推断 |
| Query / Path / Explain | 查询子图、实体最短路径和节点证据 | 更适合调用链和关系问题 |
| 可携带输出 | 生成 `graph.html`、`GRAPH_REPORT.md`、`graph.json` | 人可浏览、Agent 可消费、团队可版本化 |
| God Nodes 与设计理由 | 识别高连接节点，连接 ADR、RFC、NOTE/WHY/HACK | 发现高影响区与历史决策 |
| 20+ Agent 集成 | 适配 Codex、Claude Code、Cursor、Gemini CLI、Aider、Devin 等 | 降低跨工具迁移成本 |
| 增量更新与 Hooks | 只重建变化文件并自动保持图谱新鲜 | 避免每次全量重跑 |
| MCP stdio / HTTP | 本地或团队服务器向 Agent 暴露图查询 | 多客户端共享上下文 |
| 多格式导出 | 导出 Wiki、Obsidian、GraphML、Neo4j、FalkorDB 等 | 避免查看器锁定 |
| PR 工作台 | 汇总 CI、评审、影响社区和 PR 重叠 | 帮助安排评审优先级 |

**开箱/依赖/早期能力：** Python 3.10+；代码建图、HTML/报告/JSON、查询和增量更新可直接使用。PyPI 包名是 `graphifyy`，CLI 是 `graphify`。PDF、Office、MCP、数据库、Terraform、视频等分别需要 extras；文档和图片语义提取通常需要模型。Enterprise verification、CTO digest、Jira 等不属于当前 OSS 核心。

**同类与缺口：** GitNexus 的浏览器 Chat 和代码修改工具更完整；CodeGraphContext 更聚焦调用链；Sourcegraph Cody 的企业权限和托管体验更成熟。Graphify 的独有组合是混合资料、证据标签、可提交图文件、MCP 和广泛 Agent 集成。

**限制：** Local-first 不等于所有文档永不离机；推断关系不等于完整静态分析；约 5,000 节点后默认跳过 HTML，图文件默认上限 512 MiB；团队共享缺少细粒度 ACL/SSO；Skill/Hook 只能提示 Agent；v0.9.15 前 HTML 存在存储型 XSS；Benchmark 是项目自建的小规模测试，查询日志文档也存在冲突。

**今天关注原因：** 事实是 v0.9.15 在前一天发布并修复 XSS、嵌套 ignore 和增量更新正确性；推断是 Coding Agent 上下文成本与“快速建图 + MCP”共同推动 +1,095 日增。

来源：[README](https://github.com/Graphify-Labs/graphify)、[v0.9.15](https://github.com/Graphify-Labs/graphify/releases/tag/v0.9.15)、[Changelog](https://github.com/Graphify-Labs/graphify/blob/v8/CHANGELOG.md)、[Benchmarks](https://github.com/Graphify-Labs/graphify/blob/v8/BENCHMARKS.md)、[官方文档](https://graphify.com/docs)。

### 6. GitHub Spec Kit

**一句话定位：** 面向希望让 AI Coding Agent 按可审阅规格开发的团队，把原则、需求、计划、任务和实现收敛组织为一条跨宿主工作流。

**典型场景：** 新功能规格；遗留系统改造；多人审阅 Agent 计划；跨 Agent 迁移；把需求拆成 GitHub Issues。

| Feature | 能做什么 | 用户价值与场景 |
|---|---|---|
| `specify init` | 初始化规格目录和宿主 Agent 命令 | 降低新旧项目接入成本 |
| Constitution | 固化原则、质量门槛和不可违反约束 | 减少 Agent 反复猜测 |
| Specify | 将自然语言目标转成面向用户结果的规格 | 先对齐做什么 |
| Clarify | 主动发现歧义、边界和验收问题 | 避免不完整规格直接编码 |
| Plan | 把规格转成计划和研究任务 | 让评审发生在实现前 |
| Tasks / Issues | 生成有依赖关系的任务并转为 Issue | 提高协作和追踪性 |
| Checklist / Analyze | 检查规格、计划和任务的一致性 | 发现冲突与不可验证要求 |
| Implement / Converge | 驱动外部 Agent 实现并整理未完成项 | 从计划走向交付 |
| 约 33 个 Agent 宿主 | 支持命名宿主和通用模式 | 减少工作流锁定 |
| 暂停/恢复工作流 | 支持条件、Fan-out、Gate 与恢复点 | 让长流程可控 |
| Extensions / Presets / Bundles | 组合项目流程和治理组件 | 适配测试优先、质量门禁等规范 |

**开箱/依赖/实验：** 主流程可直接使用，但 Spec Kit 不提供模型或代码执行运行时。v0.12.14 目录中的 Memory、Test-First Governance、Autonomous Run Governance、Quality Gates 是社区 Catalog 项，不等于 GitHub 审计过的核心能力；Experimental Goals 是研究目标。

**同类与缺口：** OpenSpec 的 Delta/Archive 更适合持续规格演化；Kiro 提供更一体化的 UI；BMAD 强调专家角色。Spec Kit 的优势是宿主中立、制品透明和 Git 友好，缺点是没有 UI、模型、SDK 或完整执行环境。

**限制：** 效果依赖高级模型；Implement 容易积累大上下文；Converge 不等于测试通过；Shell 步骤并非自动沙箱化；社区扩展未经统一安全审计；不同宿主集成能力有差异。

**今天关注原因：** 事实是 v0.12.14 于前一天发布，并提升治理类 Catalog 项的可发现性；推断是团队正在从“让 Agent 写代码”转向“让 Agent 按可审阅流程工作”。

来源：[仓库](https://github.com/github/spec-kit)、[官方文档](https://github.github.com/spec-kit/)、[v0.12.14](https://github.com/github/spec-kit/releases/tag/v0.12.14)。

### 7. Marketing Skills

**一句话定位：** 面向技术营销人员、创始人和早期团队，把定位、文案、SEO、CRO、广告、分析和生命周期营销整理成可移植的 Agent Skills。

**典型场景：** 制定营销计划；审查落地页和漏斗；规划 SEO/内容；设计广告与实验；建立持续营销监测循环。

| Feature | 能做什么 | 用户价值与场景 |
|---|---|---|
| 46 个模块化 Skills | 把营销任务拆成可单独安装和调用的工作流 | 避免一个超长万能 Prompt |
| Product Marketing Context | 在 `.agents/product-marketing.md` 保存产品、客户、定位、竞品和异议 | 后续输出共享同一业务上下文 |
| 营销战略与计划 | 根据目标、阶段、渠道和资源形成有取舍的计划 | 把点子变成行动优先级 |
| 文案和内容 | 生成页面、CTA、邮件、冷邮件、社媒和内容草稿 | 提高输出结构与一致性 |
| 统一 CRO 诊断 | 分析页面、注册、激活、弹窗、表单和付费墙 | 形成更有依据的转化假设 |
| SEO / AI Discovery | 覆盖 SEO 审计、AI 搜索、程序化 SEO、站点结构和 Schema | 统一发现与内容规划 |
| 广告与创意规划 | 设计 Campaign、渠道、创意角度和测量方案 | 在投放前补齐策略 |
| 分析、埋点和实验 | 生成事件、UTM、GA4/GTM 和 A/B 测试方案 | 减少无法回答业务问题的埋点 |
| 生命周期与 RevOps | 覆盖邮件、流失挽回、推荐、销售支持和线索流转 | 把获客后环节纳入同一体系 |
| 43 个 Marketing Loops | 定义周期、触发、自检、状态和停止条件 | 把一次性建议变成持续流程 |
| 状态与 Guardrails | 提供去重、冷却、运行日志、预算上限、人工批准和 Kill Switch | 降低重复发送与错误花费 |

**开箱/依赖/实验：** 指令、模板、共享上下文和循环规格可直接使用；发布内容、读取 GA4、发送邮件、操作广告、更新 CRM 与调度需要外部工具和授权。v2.6.0 是 46 Skills；当前 main 已出现第 47 项 `marketing-council`，不能倒算进该 Release。

**同类与缺口：** 相较 n8n，领域知识更强但没有执行、调度和重跑；相较 HubSpot Breeze，开放且跨宿主但没有 CRM 原生上下文；相较 Jasper，易于 Git 管理但缺少品牌资产、审批和团队 UI。支持多种安装方式，没有自身 UI、统一 API/SDK 或端到端托管 Demo。

**限制：** 46 Skills 是说明，不是 46 个独立 Agent；43 Loops 是规格，不是已部署任务；Self-running 依赖宿主调度、权限、工具和状态；Guardrails 是书面规则，不是强制策略引擎；营销效果高度依赖真实数据，无法保证转化率或 ROAS。

**今天关注原因：** 事实是 v2.6.0 确认 46 Skills，并新增包含 43 个循环规格、九部分循环结构和七例 eval 的 Marketing Loops；推断是跨 Agent、纯 Markdown、可进 Git、带停止条件的业务循环非常符合当前趋势。

## 今天最值得体验的 3 个项目

1. **[Graphify](https://github.com/Graphify-Labs/graphify)**：体验 `query/path/explain`、关系证据标签、HTML 项目地图和 Coding Agent MCP。建议在自己的中型混合仓库验证准确性，并使用 v0.9.15。
2. **[Vibe-Trading](https://github.com/HKUDS/Vibe-Trading)**：体验跨市场研究、回测 Run Card、Alpha Zoo 和 Trade Journal/Shadow Account。只从公开数据、模拟和历史回测开始，不连接实盘账户。
3. **[AIRI](https://github.com/moeru-ai/airi)**：体验 Live2D/VRM、语音闭环、桌面伴侣、屏幕视觉和 MCP 工具。不要把 WIP 记忆和游戏能力当成熟功能。

如果目标是学习而非部署，首选 [awesome-llm-apps](https://github.com/Shubhamsaboo/awesome-llm-apps)；如果只想快速改善 AI 生成页面，可先试 [Hallmark](https://github.com/Nutlope/hallmark)。

## 值得持续跟踪的新 Feature 方向

- **有状态、幂等、带停止条件的常驻 Agent**：持续执行开始重视 Watermark、冷却、去重、人工 Gate 和 Kill Switch。
- **Agent Skills 的跨宿主包管理**：同一套业务能力跨 Codex、Claude、Cursor 复用，可能逐渐形成插件生态。
- **证据和可追溯上下文**：知识图谱、规格制品、来源标签和决策记录将补充纯向量 RAG。
- **领域 Shadow/Digital Twin**：从回答领域问题扩展到分析用户自身历史行为与长期状态。
- **Generative UI 与具身角色**：共享状态 UI、语音、视觉、桌面角色和游戏连接会继续融合。
- **本地处理与选择性云增强**：隐私敏感数据留本地，高价值语义任务按需调用云模型。
- **Agent 工作流治理组件化**：质量门禁、测试优先、审批、预算和合规将以 Preset、Skill、Extension 形式复用。

## 结论

今天的热点不是“又一个通用聊天机器人”，而是让 Agent 在真实工作中更容易安装、更懂长期上下文、更可复核、也更能持续运行。Graphify 代表上下文基础设施，Spec Kit 代表过程治理，Marketing Skills 和 Hallmark 代表跨宿主业务技能，Vibe-Trading 与 AIRI 则展示了领域 Agent 和具身交互的产品化方向。

值得保持警惕的是：很多项目把路线图、宿主能力或外部连接器写得像项目自身功能。评估时应优先验证三件事——当前 Release 是否真的包含、是否需要外部付费/授权、有没有可复现 Demo。
