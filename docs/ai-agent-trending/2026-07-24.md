## 今日 Feature 趋势摘要

1. **Agent 正从“单人终端工具”进入共享工作空间。** 今日 #1 Buzz 把 Agent 作为频道成员，#5 ego lite 给每个 Agent 独立浏览器 Space，#10 pi-web 把 Coding Agent 会话、文件和 worktree 放进 Web UI；共同主题是让人能看见、接管并审计 Agent 的持续工作。
2. **可安装 Skill 正成为垂直能力交付物。** `awesome-claude-skills` 用目录覆盖通用工作流，`text-to-cad` 则把 CAD、机器人描述、切片和制造交接做成专用 Skills；热点从“提示词集合”向带工具、文件产物、验证步骤的可执行工作流迁移。
3. **可靠性与成本治理成为 AI 工具的主 Feature。** OmniRoute 强调多 Provider 路由、额度与 fallback，OpenCodeReview 强调用确定性覆盖/定位减少 Agent 漏审和漂移；它们卖的已不只是“能调用模型”，而是可控失败、可控费用和可重复结果。
4. **Local-first 同时服务隐私、登录复用和低延迟。** Kronos 本地预测、RuView 边缘感知、pi-web 本地读会话、ego lite 复用本机浏览器登录、Buzz 可自托管；但“本地”不等于零硬件、零模型下载、零外部数据源或零配置。
5. **高密度可视 Demo 仍是最强传播器。** World Monitor 的地图、RuView 的“WiFi 无摄像头感知”、text-to-cad 的 3D 产物、ego lite 的并行 Space 都能在短视频/GIF 中直接展示价值，比抽象 SDK 更容易在 Trending 获得增量。
6. **热度与成熟度继续错位。** 今日同时出现有正式桌面包/Release 的应用、无 Release 的研究模型与清单、需要真实硬件标定的感知平台，以及 README 明确标注 coming soon/experimental 的能力；“能看到 Demo”不能自动推导为生产可用。

### 筛选说明

- `block/buzz` 的一句简介很抽象，但 README 明确将其定义为人类与 AI Agent 共处的自托管工作区，并区分“Works today / Being wired up / pending code”，因此纳入并按该状态表核验。
- `RuView` 不是 LLM/Agent 产品，但其核心直接包含预训练模型、本地学习、姿态/活动/生命体征推理和 Agent/智能家居消费接口，符合“多模态、模型推理、AI 基础设施”口径；必须把模拟数据、研究指标和真实硬件效果分开。
- `awesome-claude-skills` 是目录与样例集合，不是 Agent Runtime；因其直接交付可安装 Skills/Plugins 和外部应用工作流而纳入，但不把目录规模等同逐项生产验证。
- `Pumpkin` 虽有 “Entity AI” tracking issue，核心仍是 Minecraft 服务器；`Harper` 虽处理自然语言，README 明确强调小型、离线语法检查而非 LLM/Agent；两者均不因宽泛的 “AI” 字样纳入。

## Feature 深度卡片

### 1. [Buzz](https://github.com/block/buzz)

**一句话定位：** 面向软件团队的自托管人机协作工作区，让人、AI Agent、频道、Git 活动和自动化共享可检索、可审计的项目记录；它是 Agent 的协作面，而不是另一个单体编码 Agent。

**典型场景：** 把 Codex/Claude Code/Goose 当频道成员；将分支、评审与讨论放进同一项目房间；按消息/reaction/schedule/webhook 触发工程自动化；事故历史检索；在内网部署人机协作空间。

| Feature | 能做什么 → 用户价值 → 适用场景 | 状态 / 边界 |
|---|---|---|
| **Agent 独立签名身份** | Agent 以自己的 keypair 加入频道、线程和 DM → 可区分真人/Agent 责任并完整复盘 → 代码评审、事故频道 | **可用**；入站作者可设 owner/allowlist/anyone/nobody，[README](https://github.com/block/buzz#what-is-this-really) |
| **ACP Agent 接入** | 把频道 mention 交给 Goose、Codex、Claude Code 或兼容 ACP runtime → 复用现有 Agent 且不锁供应商 → 团队问答/编码 | **需配置** relay、私钥、适配器与模型 key；断线重连/未处理 mention 重放已有，[buzz-acp](https://github.com/block/buzz/blob/main/crates/buzz-acp/README.md) |
| **Agent-first JSON CLI** | 以稳定 JSON/退出码操作消息、频道、成员、canvas、workflow、repo、memory、上传 → Agent/CI 不必点 GUI → 自动化与脚本 | **可用但需 relay/身份**；当前安装偏源码，[buzz-cli](https://github.com/block/buzz/blob/main/crates/buzz-cli/README.md) |
| **频道级并行 Worker** | 启动 1–32 个 Agent 子进程，不同频道并行、同频道串行，并以 heartbeat 检查待办 → 增吞吐又减少同频道踩踏 → 多项目支持 | **可用但需调参**；worker 共享可见身份、资源线性增长、跨频道无顺序保证，[并行说明](https://github.com/block/buzz/blob/main/crates/buzz-acp/README.md#parallel-agents--heartbeat) |
| **多模型/本地模型入口** | 最小 `buzz-agent` 可接 Anthropic、OpenAI、Databricks、Ollama/vLLM/llama.cpp 等兼容端点 → 在成本、隐私和效果间选择 → 私有模型/混合云 | **需外部服务或本地推理**；无默认免费模型，本地模型须支持 tool calling，[Providers](https://github.com/block/buzz/blob/main/crates/buzz-agent/README.md#providers) |
| **Git 项目与评审面** | 浏览 repo/branch/tag、看 diff、行内评论、评审决定和冲突恢复，并记录 patch/status 事件 → 代码与决策不脱节 → Agent 补丁评审 | Git events/hosting **可用**，桌面 Projects 仍 **Preview**；不能当成熟 GitHub/GitLab 替代，[v0.4.22](https://github.com/block/buzz/releases/tag/v0.4.22) |
| **YAML 事件工作流** | 用消息、reaction、schedule、webhook 触发流程，CLI 管理 runs → 自动化与频道讨论共享记录 → 发布/值守 | triggers **可用**，桌面 Workflows **Preview**，approval gates 仍在接线，[状态表](https://github.com/block/buzz#works-today--being-wired-up--strong-opinions-pending-code) |
| **搜索、审计与项目记忆** | 全文查消息/作者/时间、查看审计，Agent 可维护 relay memory → 事故回答能回链历史依据 → 复盘/交接 | FTS、audit、memory CLI **可用**；“带 receipts 回答”仍需 Agent/LLM 组合，不是零配置向量 RAG |
| **Persona / Team Pack** | 版本化多个 Agent persona、共享说明和 Skills，并校验/导入 → 快速复用审查角色组合 → 多 Agent 评审 | pack 工具/示例**可用**；不是通用 agent-to-agent router 或自动 fan-out，[示例](https://github.com/block/buzz/tree/main/examples/meadow-core) |
| **桌面包与自托管** | 提供 macOS/Linux/Windows 包、Compose 与 Helm → 可从本地 PoC 走向内网部署 → 团队试点 | OSS 客户端默认连 localhost，仍需自建/指定 relay；Helm 内置依赖仅建议评估，[v0.4.24](https://github.com/block/buzz/releases/tag/v0.4.24) |

**易用性与同类差异：** 桌面 UI、JSON CLI、REST/WS、ACP、模型适配、YAML、Persona Pack 和部署材料齐；相较 Mattermost + Agents，Buzz 独有 Agent 独立签名身份以及聊天/Git/workflow 共用事件与审计面，Agent/CLI 更一等。Mattermost 则在移动端、通知、Playbooks、人类侧权限/治理、现成摘要与 RAG 上更成熟。

**限制与关注原因：** `buzz-agent` 无持久 session、无图片/音频 prompt，MCP 仅 stdio；Projects/Workflows/Forum/Pulse 等入口仍 Preview，mobile、huddle lifecycle、approval glue 正在接线，push 与跨 relay reputation 只是愿景。事实：今日 #1、+2,162，刚发布 `v0.4.24`。推断：Block 背景、诚实的状态表、Agent 作为“一等同事”的强叙事，以及可见的桌面/Git/工作流 Demo 共同推动上榜。

**典型场景：** 突发事件跨源核验；国家风险快读；物流/商品暴露研判；个人持续监控；给 Claude、Cursor 或内部 Agent 提供实时世界数据。

|---|---|---|
| **56 类实时态势层** | 同屏看冲突、制裁、军设、海缆、管线、AIS、机场、火灾等 → 在地理上下文中发现关联 → 值守/安保 | **免费可试**；通常 5–15 分钟刷新，部分层需外部 key，[Features](https://www.worldmonitor.app/docs/features) |

| **本地边缘与自适应** | 板端运行 presence/vitals/fall，服务器用本地样本训练房间分类器 → 降上云与固定阈值成本 → 隐私场景 | 可用但每类需录 30–60 秒，换布局/节点需重录；高级姿态/计数/记忆仍需 host/额外模型 |
| **UI + API + CLI + Python** | Three.js/Observatory、热图、REST/WS、Rust CLI、PyPI client → 从 Demo 到集成路径清晰 → 可视调试 | 软件面易试；point-cloud 可选摄像头/MiDaS，开启后不再是纯无摄像头，远程暴露需自配 TLS/auth |
| **Home Assistant / Matter** | MQTT 自动发现发布原始+10 个语义状态和 Blueprints → 快速进家居自动化 → 占用/照护 | HA MQTT 有 broker 集成测试；Matter 在 v0.7.0 明确仅 scaffolding、wiring 延后，当前“扫码即用”文案不采信，[范围说明](https://github.com/ruvnet/RuView/blob/main/docs/releases/v0.7.0-mqtt-matter.md) |
| **`@ruvnet/rvagent` MCP** | Agent 查询 CSI、存在、呼吸/心率、pose/count Cog 和训练任务 → 不必重写传感器客户端 → Ambient Agent | `v0.2.0` 已发布，需 Node 20+ 与后端；policy/vector/semantic 等仍 roadmap，模拟后端也会返回模拟答案，[npm](https://www.npmjs.com/package/@ruvnet/rvagent) |

**易用性与同类差异：** 预编译固件、Docker、详细指南、DEMO/LIVE 标记、UI、REST/WS、Python、npm MCP、HA Blueprints 和 HF 权重构成罕见的“传感器到 Agent”纵向链。相较 Espressif ESP-CSI，它上层场景更丰富但芯片/基础采集边界不如厂商示例清晰；相较商用 mmWave/ESPHome，研究面更广，却远不如固定区域 presence 即插即用。

**限制与关注原因：** 真实多人/多房间数据、默认 learned pose 路径、医学/跌倒验证和 Matter 配对仍是缺口；换环境需重标；无摄像头仍能推断敏感状态，需要告知与访问控制。项目已在 [Proof of Capabilities](https://github.com/ruvnet/RuView/blob/main/docs/proof-of-capabilities.md) 撤回旧“100% presence”、94.2% pose、96.5% fall 等口径，这是积极信号。事实：今日 #12、+1,708，最新 [v1954](https://github.com/ruvnet/RuView/releases/tag/v1954) 只换宣传图/alt text，没有新增感知 Feature。推断：WiFi“穿墙”、低成本 ESP32、隐私叙事与高频提交放大传播；真正价值是降低 CSI 原型门槛，而非已经成为医疗雷达。

## 今天最值得体验的 3 个项目

1. **[Buzz](https://github.com/block/buzz)：最值得验证“Agent 是团队成员”是否真的好用。** 最吸引人的不是聊天，而是独立 Agent 身份、ACP 接入、JSON CLI，以及频道/Git/workflow 共用审计记录。建议用本地 relay + 一个测试 Agent 跑“频道提问 → 补丁 → 人审”的最小闭环，同时主动避开 Preview 的审批/Projects 全量承诺。
2. **[World Monitor](https://github.com/koala73/worldmonitor)：五分钟内最容易感知价值。** 无需注册即可看多层地图、新闻聚类、国家风险和基础 Brief；若要验证 Agent 价值，再查看 MCP 目录并确认 Pro/API 限制。最吸引人的 Feature 是“多源现实信号 → 可视态势 → AI 解释 → Agent 工具”的连续链。
3. **[OpenCodeReview](https://github.com/alibaba/open-code-review)：最容易在开发工作流中做 A/B。** 先用 `review --preview` 与 `rules check` 零成本确认覆盖，再拿一个小 commit 输出 JSON、打开 Viewer 看行定位与工具调用；有 Codex/Claude 订阅可直接比较 delegation 与独立模型配置。最吸引人的 Feature 是覆盖预览、按文件规则、resume/Viewer 和 CI 行评论的组合。

## 值得持续跟踪的新 Feature 方向

1. **有持久身份和权限边界的 Agent Workspace。** Buzz 把 Agent 从 webhook 升为签名成员；下一步真正关键的是审批、移动端、通知、长期 session、跨 Agent 协同与企业权限，而不只是更多聊天 UI。
2. **从“说明型 Skill”走向“带验证器与安全契约的 Skill”。** CAD Skills 的几何检查、dry-run 和 `need more info`，以及 OpenCodeReview 的 preview/rules，比大量只含 `SKILL.md` 的目录更接近可靠复用。值得跟踪统一依赖、版本、权限清单、签名与可复现验收。
3. **可接管的 Agent Browser Workspace。** ego lite 的 Space 说明“人继续工作、Agent 在旁边跑”正在替代抢焦点自动化；未来竞争点会是跨平台、Profile/Session 安全、无头/CI、代理、MCP/ACP 和版本兼容，而非只比 Snapshot 口号。
4. **Agent 的预算、覆盖与失败契约。** OmniRoute 管额度/fallback，OpenCodeReview 暴露 preview/session/telemetry；下一步应有审计级 coverage manifest、全局 Token/工具调用预算、partial-success 语义和可重复 eval。
5. **现实世界数据源的 provenance 与决策分层。** World Monitor 和 RuView 都把现实信号送给 Agent；必须同时输出 source/freshness、模拟/实采、模型/规则、精度/校准和“仅供提醒/不可诊断”的机器可读边界。
6. **Ambient Agent 的本地多模态感知。** WiFi CSI、毫米波、音频和设备事件将成为 Agent 的环境输入；真正的产品拐点不是再加一个 MCP tool，而是跨房/跨人验证、知情同意、隐私门控和安全冗余。
