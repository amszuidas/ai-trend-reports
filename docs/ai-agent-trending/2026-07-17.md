# Feature 深度卡片

## 1. Apache Ossie

**一句话定位：** 为数据平台、BI/AI 工具厂商和数据团队提供厂商中立的语义模型交换规范与参考转换工具，减少指标、字段和业务定义在不同系统间漂移；它不是查询引擎、BI 产品或现成 Agent。

**典型场景：** 在 dbt、GoodData、Omni、Salesforce 等语义格式间迁移；以 Git 统一维护指标口径；给数据 Agent 提供结构化业务语义；同时保存多仓库 SQL 方言；在 CI 校验模型。

### 最值得关注的 Features

1. **中立 JSON/YAML 规范** → 用机器可读文档承载语义模型 → 便于版本控制、Review 和跨工具迁移 → 多 BI/数据平台共存。**状态：0.2.0.dev0 Draft，需固定版本。** [核心规范](https://github.com/apache/ossie/blob/main/core-spec/spec.md)
2. **统一 Dataset/Field/Relationship/Metric** → 描述来源、主键、计算字段、外键和聚合指标 → 避免在每个下游重建 Revenue 等口径 → 指标治理、Agent 数据问答。**状态：核心可用；复杂关系、粒度和累计指标仍不足。**
3. **分层 AI Context** → 在模型、字段、关系、指标上附 instructions、synonyms、examples → 给 LLM 明确业务叫法和限制 → Text-to-SQL、指标发现。**状态：Schema 可用，但 Ossie 不运行 Agent；下游必须消费。**
4. **多方言表达式** → 同一字段/指标保存 ANSI、Snowflake、Databricks、BigQuery、MDX、Tableau、MAQL 等表达 → 减少跨平台重写 → 多仓库部署。**状态：基础可用；`Ossie_SQL_2026` 仍为 Proposed Final。** [Expression Language](https://github.com/apache/ossie/blob/main/core-spec/expression_language.md)
5. **Vendor Custom Extensions** → 保存核心规范无法表达的厂商 JSON → 提高同厂商导入后再导回的保真度 → 渐进迁移。**状态：可用；其他工具通常只能透传，不能执行。**
6. **本地 Validator** → 校验 JSON Schema、重复名称、关系引用和部分 SQL 语法 → 把明显错误前移到 CI → 模型治理。**状态：可运行；需 Python 3.11 等依赖，MDX/Tableau/MAQL 不做语法验证。** [Validator](https://github.com/apache/ossie/blob/main/validation/validate.py)
7. **参考转换器集合** → 覆盖 dbt、GoodData、Honeydew、Omni、OrionBelt、Polaris、Salesforce 双向转换，以及 Ossie→Snowflake Cortex Analyst → 让标准交换可真实试验 → 平台迁移。**状态：成熟度和入口不一。** [Converters](https://github.com/apache/ossie/blob/main/converters/README.md)
8. **显式降级与回转保真** → 转换器以 warning/error 或 extension 处理无法映射能力 → 用户能识别迁移是否等价 → 迁移审计。**状态：不保证跨所有格式无损；stash 不等于目标平台可用。**
9. **Python 类型与示例** → 提供 Pydantic 类型、TPC-DS、Flights Ontology 和转换 fixtures → 降低工具作者读写规范的门槛 → 新转换器、兼容测试。**状态：开发者可用；无统一 UI/Demo。** [Python package](https://github.com/apache/ossie/tree/main/python) · [TPC-DS](https://github.com/apache/ossie/blob/main/examples/tpcds_semantic_model.yaml)
10. **Ontology 方向** → 描述业务概念、值类型、继承、关系和约束 → 有机会把“表结构”提升为 Agent 可消费的企业概念层 → 金融、企业知识。**状态：实验/开发中，不能视为稳定生产能力。** [Ontology](https://github.com/apache/ossie/blob/main/ontology/ontology.md)

**同类与缺口：** 相比 dbt MetricFlow，Ossie 强在中立交换、多方言和扩展透传，弱在复杂可执行指标；相比 Cube，它不要求采用同一运行产品，但缺查询 API、权限、缓存、UI 和运行时。Roadmap 中的统一 CLI、Registry、Sync API、查询语言、参考执行引擎、verified queries 和 AI 暴露控制都不能写成已交付。[Roadmap](https://github.com/apache/ossie/blob/main/ROADMAP.md)

**易用性与限制：** YAML/JSON、Validator、Python 类型、转换器和示例齐全，但转换器打包方式不统一，部分需 Java/Maven、外部服务或凭据。“Write Once, Query Anywhere”目前主要是格式与尽力转换；`ai_context` 可辅助 grounding，但没有内置 LLM、MCP 或准确率保证。项目处于 Apache Incubator、无正式 GitHub Release。

**为何今天关注：** 事实是项目 7 月 10 日进入 Apache Incubator并完成 OSI→Ossie 更名，本周又合入 OrionBelt/Omni 转换器和表达式文档。[官方公告](https://ossie.apache.org/updates/ossie-enters-apache-incubator/) 推断上，Apache 治理曝光与“给 Agent 统一业务语义”的需求共同推动了榜单热度。

## 2. Hallmark

**一句话定位：** 为 Claude Code、Cursor、Codex 用户提供“反 AI 模板感”的前端设计 Skill，通过结构轮换、审计、重设计和自检门禁，让页面更像为具体品牌设计，而不是同一 SaaS 模板换色。

**典型场景：** 从 Brief 构建品牌页；审计现有 UI；保留业务边界做重设计；从截图/URL 提取设计 DNA；为多页面项目固化 Tokens 和 `design.md`。

### 最值得关注的 Features

1. **结构优先 Build** → 先选约 21 种页面宏结构，再匹配 20 个主题 → 真正改变页面骨架而非只换色 → Landing Page、作品集、开发工具官网。**状态：安装 Skill 后可用，质量依赖宿主模型。** [Macrostructures](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/references/macrostructures.md)
2. **六轴自评 + Anti-slop 门禁** → 检查设计意图、层级、具体性、响应式、对比度、诚实文案等 → 减少模板化 Hero、虚构指标和移动端溢出 → 交付前自检。**状态：规则可用，但不是确定性 Linter；当前实际为 1–57 加 38a，README 的 57 已漂移。** [Slop Test](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/references/slop-test.md)
3. **`hallmark audit`** → 输出 critical/major/minor 问题、文件位置和修复建议，不改代码 → 先诊断再决定是否改版 → UI Review、设计债盘点。**状态：开箱可用。** [Audit](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/references/verbs/audit.md)
4. **`hallmark redesign`** → 保留路由、文案意图、品牌、信息架构和业务逻辑，重做结构、节奏和交互 → 降低“重设计等于重写应用”的风险 → 已有产品翻新。**状态：可用；大规模删除和多页方向需确认。** [Redesign](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/references/verbs/redesign.md)
5. **`hallmark study`** → 从截图/URL提取宏结构、组件、字体角色、色彩锚点和视觉节奏，可导出 `design.md` → 借鉴设计语言而非像素克隆 → 品牌研究。**状态：需宿主视觉/网页读取能力；付费字体需授权。** [Study](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/references/study.md)
6. **Pre-flight 项目识别** → 先识别现有字体、色板、动效库、框架、间距和 `design.md` → 降低覆盖既有 Tailwind/品牌规范的风险 → 真实仓库增量开发。**状态：需项目文件权限。**
7. **Catalog/Custom/Bespoke 三档路线** → 在目录主题、一次性 OKLCH 方案和完全定制之间选择 → 平衡速度、一致性和个性 → 强品牌项目。**状态：Custom 已可用；外部素材可能收费。** [Custom Theme](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/references/custom-theme.md)
8. **组件八状态** → 单组件 Brief 直接生成 default、hover、focus、active、disabled、loading、error、success 预览 → 一次发现状态缺漏 → Button/Input/Modal。**状态：开箱可用；不是可视化组件编辑器。**
9. **跨运行设计记忆** → `.hallmark/log.json` 记录最近结构、主题和增强选择 → 主动避开连续复用同一形态 → 多页面长期项目。**状态：规则轮换，不是视觉相似度模型。** [Skill](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/SKILL.md)
10. **设计系统导出** → 用户明确锁定方向后生成 `design.md`/`tokens.css`，并约定 Tailwind、DTCG、shadcn 形式 → 供后续页面和其他 Agent 复用 → 多页交接。**状态：需显式触发，不是默认产物。** [design.md](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/references/design-md.md)

**同类与缺口：** 相比 Anthropic frontend-design Skill，Hallmark 强在四种 verbs、结构/主题目录、项目记忆、八状态和可审计门禁；代价是规则更重、审美更有主见。它缺独立 GUI/API/SDK、自动截图视觉反馈、Figma 集成和确定性质量测试。

**易用性与限制：** `npx skills add nutlope/hallmark` 一行安装，官方 [Live Demo](https://www.usehallmark.com/)、[Recipes](https://github.com/Nutlope/hallmark/blob/main/docs/recipes.md) 和 Study 示例直观；但 URL Study、视觉输入和外部素材受宿主/授权限制。Cursor references 兼容性存在 Issue；“Anti-AI-slop”是主观约束，不是客观审美 Benchmark。图片生成、Brand-first、variant、图表规则、Explain 和 MCP Live Preview 仍在 [Roadmap](https://github.com/Nutlope/hallmark/blob/main/ROADMAP.md)。

**为何今天关注：** 事实是今日榜单第 2、单日 +3,372，且四种操作、一行安装和互动 Demo 都可立即理解。推断上，“AI 页面千篇一律”是高频痛点，鲜明定位与易展示性共同放大了传播。

## 3. PostHog（边界纳入：AI/Agent 功能子集）

**一句话定位：** 面向 AI 产品工程团队，把 LLM/Agent Trace、质量评估、真实用户行为、产品分析和修复执行放进同一工作区；PostHog 仍是综合产品平台，而非纯 AI 项目。

**典型场景：** 排查多步 Agent；核算一次任务的成本/延迟；持续评估幻觉/格式/情绪；把 Trace 与 Replay/转化关联；通过 MCP 或 Self-driving 从生产信号生成分析与 PR。

### 最值得关注的 Features

1. **端到端 AI Tracing** → 捕获 generations、spans、embeddings、trace/session，展示树形时间线、成本、延迟、token 和错误 → 还原一次 Agent 运行 → RAG、工具调用、多轮自动化。**状态：可用；需 wrapper、OTel 或手动埋点。** [Traces](https://posthog.com/docs/ai-observability/traces)
2. **Tool Analytics** → 自动抽取工具名、次数、共现组合、调用路径和趋势 → 看见 Agent 实际依赖和异常工具链 → MCP/function calling。**状态：常见官方 wrapper 接近开箱，自定义格式需标准属性。** [Tools](https://posthog.com/docs/ai-observability/tools)
3. **成本/延迟/Session 聚合** → 计算 token、缓存、搜索和 reasoning 成本，并聚合完整业务任务 → 从“单调用”提升到“完成一件事花多少” → Agent FinOps。**状态：价格是估算；私有模型和协议价需自定义。** [Costs](https://posthog.com/docs/ai-observability/calculating-costs)
4. **与产品行为关联** → Trace 可链接 Session Replay、Error Tracking、漏斗和用户群 → 判断模型指标是否真正影响体验和转化 → AI 搜索、客服、Copilot。**状态：需传播 session/trace ID，并接入各采集产品。**
5. **在线 Evaluations** → LLM Judge、Hog 规则和情绪模型对采样 generation 持续评分 → 把人工抽查变成质量信号 → 幻觉、格式、安全、语气。**状态：可用；Judge 需第三方模型 Key/费用。** [AI Evals](https://posthog.com/docs/ai-evals)
6. **Prompt Management + Experiment** → 管理变量、不可变版本、Diff、回滚和 SDK 缓存，并以成本/延迟/Eval 做 A/B → Prompt 变化可追溯且可用真实流量验证 → PromptOps。**状态：Beta；有 1MB/2,000 版本等限制。** [Prompt Management](https://posthog.com/docs/prompt-management)
7. **Playground 与 Trace 重放** → 并排比较模型、参数、工具调用，并从生产失败样本一键重放 → 缩短复现到迭代路径 → 模型选型、Prompt 调优。**状态：基础模型可试；更多模型/额度需 BYOK。**
8. **Trace 摘要/聚类/情绪** → 生成流程图与关键点、聚类相似交互、标记负面会话 → 从大量对话发现系统问题 → Agent 运营。**状态：Summary Alpha；Clusters/Sentiment Beta，且有 AI 数据处理授权和限额。** [AI Trace 分析](https://posthog.com/docs/ai-observability/analyze-traces-ai)
9. **PostHog AI** → 用自然语言生成 SQL、Insight、Dashboard、Flag、Experiment、Survey 并总结 Replay/实验 → 降低产品分析门槛 → 转化研究、实验复盘。**状态：Cloud/AI Credits；Plan/Research Mode 为 Closed Beta。** [PostHog AI](https://posthog.com/docs/posthog-ai)
10. **Hosted MCP** → 让 Codex、Claude、Cursor 查询 analytics/traces，也可创建 Flag、实验、Survey、CDP 和 Workflow → 把生产上下文带入编辑器 → 从错误到 kill switch。**状态：Beta；应优先 read-only/最小工具集合。** [MCP](https://posthog.com/docs/model-context-protocol)
11. **Self-driving 闭环** → Signals 聚合成报告，Agent 在沙箱调查并创建待审 PR，合并后继续测量 → 将维护问题主动送到工程队列 → 死点击、埋点缺口、重复报错。**状态：Open Beta；每月 3 PR 免费后 $15/PR，不能自动 merge。** [Self-driving](https://posthog.com/docs/self-driving)
12. **PostHog Code** → 计划把生产上下文、多 Agent 和编码执行放进桌面入口 → 让编码 Agent 看见真实用户问题 → 生产驱动修复。**状态不确定：Self-driving 页写 Beta/Get app，专属页同时写 launching soon/waitlist，应按限量 Beta/候补处理。** [PostHog Code](https://posthog.com/code)

**同类与缺口：** 相比 Langfuse/Braintrust，PostHog 的强项是 Trace 与 Replay、漏斗、Survey、Flag、Experiment、Warehouse 同一用户语境，并进一步延伸到 MCP 和 PR；不足是 Prompt、聚类、Datasets/离线回归等专项成熟度不均，Braintrust 的 Dataset/CI Eval 流程更完整。

**易用性与限制：** Web UI、SDK/OTel 接入矩阵、Wizard、Playground、MCP 用例和 Demo 都很强；Cloud AI Observability 前 100K events/月免费，PostHog AI 每月 500 credits 免费。大字段只保留 30 天；AI 数据会发往模型供应商；Privacy Mode 会削弱内容级调试。“开源平台”不等于全部 AI 能力可 FOSS 自托管，PostHog AI 明确 Cloud-only。[定价](https://posthog.com/pricing)

**为何今天关注：** 事实是 README 已将 Self-driving 和 AI Observability 放在核心定位，MCP 与 Open Beta PR 闭环均有公开文档。推断上，热度来自“产品分析 + AI 质量 + Agent 执行”一体化的认知跃迁，而非已核实的单日重大 Release。

## 4. Open Interpreter

**一句话定位：** 为希望用 Kimi、Qwen、DeepSeek、Claude、OpenAI 或本地模型编码的开发者，提供统一 Codex 风格终端、模型适配 Harness 和安全控制。

**版本边界：** 本卡片只描述新 Rust [v0.0.28](https://github.com/openinterpreter/openinterpreter/releases/tag/rust-v0.0.28)。旧 Python 项目的 `pip install`、`interpreter.chat()`、通用媒体处理等能力已移交社区维护，不能算入当前产品。

**典型场景：** 仓库内修 Bug/重构/测试；用低成本模型做 coding agent；只读 Review；通过 ACP/Codex SDK/MCP 嵌入编辑器和服务；操作 Web/桌面应用做回归验证。

### 最值得关注的 Features

1. **Codex 风格编码循环** → 读仓库、编辑文件、运行命令、Review diff 并续接会话 → 避免在聊天与终端搬运上下文 → 日常开发。**状态：安装后可用；仍需模型。** [Quickstart](https://www.openinterpreter.com/docs/terminal/quickstart)
2. **多 Harness 切换** → `/harness` 选择 native、Claude Code、Kimi Code、Qwen Code、DeepSeek TUI、SWE-agent 等 → 给不同模型更贴近其推荐的工具和交互形态 → 模型比较。**状态：可用，但 wire API 有严格兼容矩阵；不是外部 CLI 本体。** [Harness](https://www.openinterpreter.com/docs/terminal/harness)
3. **Kimi K3 原生体验** → 自动使用 `kimi-code` Harness、保留 thinking history 和默认 Skills → 无需另装 Kimi Code CLI → 长上下文编码。**状态：0.0.27/0.0.28 当天新发，非常早期；需 Kimi 订阅/API。** [Kimi K3](https://www.openinterpreter.com/docs/terminal/kimi-k3)
4. **Provider/模型切换** → `/model` 支持托管商、Ollama、LM Studio 和兼容端点 → 按成本、隐私和质量换后端 → 成本控制、本地推理。**状态：选择器可用；外部服务和本地 Runtime 需另配。** [Providers](https://www.openinterpreter.com/docs/terminal/providers)
5. **Sandbox + Approvals** → 提供 read-only、workspace-write、danger-full-access 与独立审批策略 → 分开控制执行边界和确认时机 → 陌生仓库审计、CI。**状态：跨 OS 强制能力不同；`--yolo` 会同时移除保护。** [Sandbox](https://www.openinterpreter.com/docs/terminal/sandbox)
6. **ACP 接入** → `interpreter acp` 为 Zed、JetBrains 等客户端提供会话、流式进度和权限请求 → 在 IDE 中获得结构化体验 → 编辑器 Agent。**状态：需客户端配置；VS Code 依赖社区扩展。** [ACP](https://www.openinterpreter.com/docs/terminal/acp)
7. **Codex SDK/App Server 兼容** → 既有 SDK 只需替换 binary 路径或启动 `interpreter app-server` → 保留 thread、stream、approval 调用 → 内部平台迁移。**状态：可用但无独立 SDK；兼容测试不保证未来永久一致。** [SDK](https://www.openinterpreter.com/docs/terminal/sdk)
8. **双向 MCP** → 既能连接 stdio/HTTP MCP，又能作为 `mcp-server` 被其他 Agent 调用 → 接企业工具或成为 Agent-of-agents 的编码工具 → 自动化平台。**状态：协议能力可用；服务、凭据与费用另配。**
9. **Skills/AGENTS.md** → 加载团队工作流和项目上下文；0.0.28 特别把 Open Interpreter Skills 带入 Kimi Code 模式 → 重复流程可版本化复用 → 发布、迁移、评审。**状态：Skills 可用；Kimi 接入为当天新能力。** [Skills](https://www.openinterpreter.com/docs/terminal/skills)
10. **Computer Use QA** → bundled QA Skill 调用 `agent-browser`/`trycua` 操作 Web/桌面并验证前后状态 → Agent 不再只“写完宣称成功” → 登录、表单、状态回归。**状态：需安装外部驱动、网络和权限；不是内建离线浏览器。**
11. **`exec` 结构化自动化** → 支持 stdin、diff、图片、JSONL/JSON Schema、resume/review/verify → 便于 CI 和机器消费 → PR Review、批处理。**状态：开箱可用；CI 应用窄权限。** [Exec](https://www.openinterpreter.com/docs/terminal/exec)
12. **可恢复会话与 Subagents** → resume/fork/compact，并派生独立线程做搜索/评审 → 长任务可续跑且隔离调查 → 大仓库。**状态：当前 build 默认多 Agent；额外线程增加费用。** [Sessions](https://www.openinterpreter.com/docs/terminal/sessions)

**同类与缺口：** 相比 Codex CLI，它增加 Provider picker 和多 Harness；相比 Kimi/Qwen/Claude 官方 CLI，它提供统一入口，但 Harness 只是重实现的产品形状，不是官方完整兼容。缺口包括独立 SDK、稳定 1.x 版本，以及公开性能 Benchmark。

**易用性与限制：** 有 TUI、`exec`、ACP、App Server、MCP、CI 示例和跨平台 Release；文档覆盖完整。但 README 的“maximum K3 performance”没有独立对照数据；Computer Use 依赖外部工具；本地模型质量与硬件差异大。今天连续发布 0.0.27/0.0.28，Release 正文为空，变化需 tag compare 核验。

**为何今天关注：** 事实是今天的 0.0.27 加入 Kimi K3/Kimi Code，0.0.28 又把 Skills 接入 Kimi 模式。[0.0.27→0.0.28](https://github.com/openinterpreter/openinterpreter/compare/rust-v0.0.27...rust-v0.0.28) 推断上，“新旗舰模型 + 推荐 Harness + 熟悉 Codex UI”正好满足降低 coding-agent 成本又不想重做工具链的需求。

## 5. Bonsai-demo

**一句话定位：** 为希望在笔记本、单卡乃至手机内存预算中运行较强本地模型的开发者，把低比特 Bonsai 27B、视觉、推理强度、工具调用、MCP 和 Web Demo 打包成可体验的本地多模态 Agent。

**典型场景：** 隐私敏感本地问答；消费级硬件上的 27B Agent；用 OpenAI-compatible API 替换后端；在 Open WebUI 演示 SQL/Python/搜索工具；测试长上下文和低内存选项。

### 最值得关注的 Features

1. **1-bit/Ternary 双档 27B** → 约 3.9GB 版优先占用、约 7.2GB 版优先质量 → 按硬件与质量选档 → 笔记本、单卡、MLX Swift。**状态：权重可用；质量/速度数字主要为项目方自测。** [27B Collection](https://huggingface.co/collections/prism-ml/bonsai-27b)
2. **尺寸/格式矩阵** → 27B、8B、4B、1.7B，覆盖 GGUF/MLX → 从手机级到笔记本级缩放 → 硬件选型。**状态：可用；视觉、Thinking、原生 Agent 能力主要是 27B，不能泛化到小尺寸。**
3. **一键跨平台安装** → `setup.sh`/`setup.ps1` 下载权重和预编译 Runtime，并提供 CLI/server/Open WebUI 脚本 → 缩短首次对话路径 → Mac/Linux/Windows 试玩。**状态：不同硬件组合成熟度不一。**
4. **本地视觉问答** → 27B 接收照片、截图和普通文字图片 → 在本地分析视觉材料 → UI 检查、轻量 OCR。**状态：可用；projector 额外占用，慢后端默认压缩视觉 token；1-bit MLX 仍文本-only。** [Vision](https://github.com/PrismML-Eng/Bonsai-demo/blob/main/VISION.md)
5. **可调 Reasoning Effort** → UI/API 选择 Off/Low/Medium/High/Max 或预算 → 在响应时间与推理深度间取舍 → 简单问答、数学/代码。**状态：27B 可用。**
6. **OpenAI 风格 Tool Calls** → 标准 `tools` 输入、结构化 `tool_calls` 输出和完整结果回填 → 复用现有 OpenAI Agent 客户端 → 本地工具 Agent。**状态：27B 可用；无独立 SDK。** [Tools](https://github.com/PrismML-Eng/Bonsai-demo/blob/main/TOOLS.md)
7. **MCP Agent 循环** → llama-server UI 预配 Hugging Face/DeepWiki，并允许添加 HTTP MCP → 本地模型可搜索模型库和仓库 → 研究助手。**状态：需网络；stdio 要桥接，工具 Schema 会增加 Prefill。**
8. **预配置 Open WebUI Demo** → 内置天气、抓网页、示例 SQL、Jupyter/Python、市场数据和图表 → 无需手工拼接即可演示多步 Agent → 数据调查、产品展示。**状态：核心可用；Brave/Jupyter 等需 Key/环境。** [Open WebUI](https://github.com/PrismML-Eng/Bonsai-demo/blob/main/OPENWEBUI.md)
9. **262K Context** → 模型可分配超长窗口 → 减少频繁切块 → 长文/长对话。**状态：上限已发布，不等于完整窗口推理质量或低内存保证。**
10. **Prompt Cache** → llama.cpp 复用稳定前缀和图片编码 → 改善多轮视觉与固定 MCP Schema 的首字延迟 → 交互分析。**状态：llama.cpp 可用；MLX 无跨请求 Cache。**
11. **4-bit KV Cache** → 显著降低长上下文 KV 占用 → 紧内存机器容纳更长会话 → 百 K 文档。**状态：实验、默认关闭、仅 llama.cpp；省内存但可能略慢。** [KV Cache](https://github.com/PrismML-Eng/Bonsai-demo/blob/main/KV-CACHE.md)
12. **DSpark 推测解码** → CUDA 下用 drafter 加速代码/推理输出 → 缩短重推理延迟 → 单用户实验。**状态：高度实验，Mac/并发/多轮 Cache 不适用。**

**同类与缺口：** 相比普通 llama.cpp 模型，差异点是“低占用 27B 多模态 reasoning/tool-use 模型 + 预配置体验”，而 UI、API、MCP 多来自运行时；相比 LM Studio，缺桌面安装器、通用模型市场、RAG 和正式 SDK。

**易用性与限制：** CLI、Web UI、Open WebUI、OpenAI API、示例和硬件 benchmark 模板齐全。README 仍写 27B 私有时需 HF Token，但本次权重可匿名访问，需以运行时 HF gating 为准。“能放进手机内存”不等于提供成品手机 App；多页 PDF 体验证据不足；MCP/搜索会让数据离开设备；Open WebUI Demo 默认无鉴权且可执行 Python，不应直接公网部署。P40、ROCm、Metal 等仍有设备特定 Issue。

**为何今天关注：** 事实是新 27B 家族一次性加入视觉、Thinking、Tool Calling/MCP 与长上下文。[README](https://github.com/PrismML-Eng/Bonsai-demo#new-bonsai-27b) 推断上，“约 3.9GB 的 27B 还能做多模态 Agent”是醒目的传播叙事；项目适合今天试验，不宜直接按成熟生产栈采购。

## 7. LobeHub

**一句话定位：** 面向个人知识工作者、内容团队和开发者的 Agent 操作台，用可视化 UI 集中多模型、知识库、Memory、MCP、多 Agent 协作、定时执行、IM 入口及本地 Codex/Claude Code。

**版本口径：** 稳定基准为 [v2.2.10](https://github.com/lobehub/lobehub/releases/tag/v2.2.10)。默认分支 `canary` 今天已到 v2.2.11-canary.33，页面明确不用于生产；Workspace API Key/RBAC 等只能标为 canary。

**典型场景：** 研究/写作 Agent 团队；长期知识助理；定时情报；桌面统一驱动 Codex/Claude Code；把 Agent 接入 Slack/飞书/微信。

### 最值得关注的 Features

1. **Agent Builder/社区 Agent** → 自然语言生成角色、Prompt、模型和 Skills，也可一键添加模板 → 降低创建门槛 → 翻译、研究、教学。**状态：稳定；生成配置需人工检查，调用需 Credits/BYOK。**
2. **多模型/多模态入口** → 70+ Provider、不同 Agent 默认模型、本地 Ollama/LM Studio 和云图像/视频 → 按质量、成本、隐私选择 → 混合 AI 工作区。**状态：需 Provider/Key/套餐，不能理解为任意模型无条件兼容。**
3. **Agent Groups** → 顺序、并行、迭代、辩论四种协作，由 Moderator 协调总结 → 减少手工跨窗口交接 → 调研、内容流水线、决策。**状态：稳定；成员越多越慢越贵。** [Agent Groups](https://lobehub.com/docs/usage/agent/agent-team)
4. **Skills/Integrations/MCP** → 内置工具、OAuth、Composio、社区/自定义 HTTP/STDIO MCP，并按 Agent 授权 → Agent 可操作 SaaS、数据库和私有 API → 企业工具链。**状态：内置可用，外部需授权/Key；“10,000+”是目录宣传，不是审计数。**
5. **White-box Memory** → 提取结构化行为规则，支持搜索、编辑、删除和导出 → 减少跨会话重复说明且保持可控 → 长期助理。**状态：稳定，Cloud Starter+；不是模型再训练。** [Memory](https://lobehub.com/docs/usage/getting-started/memory)
6. **Resource Library/RAG** → 持久化文件、Pages、Notion ZIP 与 AI 产物，分块并语义检索 → 回答基于用户资料 → 企业知识库。**状态：稳定；受 Cloud 额度或自托管 PostgreSQL/S3/Embedding 限制。**
7. **Pages** → Markdown 编辑、预览、自动保存，Agent 可读整页并直接改写 → 消除聊天与文档复制 → 博客、方案、研究。**状态：稳定；超长文档仍受上下文限制。**
8. **Scheduled Tasks** → 按小时/日/周云端运行，有时区、次数和历史 → 无人值守监控/摘要 → 竞争情报、日报。**状态：稳定 Cloud job；不读取此前聊天，Prompt 必须自包含。** [Schedule](https://lobehub.com/docs/usage/agent/scheduled-task)
9. **Codex/Claude Code 桥接** → Desktop 启动本机 CLI、展示文件/命令/子 Agent，并支持 worktree 和会话恢复 → 用统一聊天 UX 管理编码 Agent → 并行开发、Review。**状态：稳定；需桌面、本机 CLI、登录和目录权限。**
10. **IM Gateway** → 同一 Agent 接 Discord、Slack、Telegram、LINE、QQ、微信、飞书/Lark等 → 进入现有聊天入口 → 团队机器人。**状态：稳定但平台能力不同；微信需订阅，iMessage 仍 Labs。** [Channels](https://lobehub.com/docs/usage/channels/overview)
11. **Cloud/Desktop/Mobile/自托管** → 在低运维 Cloud 与数据自主部署间选择 → 适配个人/团队 → 多交付。**状态：稳定；生产自托管仍需数据库、S3、备份、监控，Vercel 有超时/WebSocket 边界。**

**同类与缺口：** 相比 LibreChat/Open WebUI，强在 Groups、Pages、Schedule、IM 与编码 Agent 组合；相比 Dify，长期个人工作区更强，但缺通用节点式 DAG、确定性变量流和成熟应用 API。稳定版也未见通用 Agent REST API；Workspace API Key/RBAC 仍在 canary。

**易用性与限制：** Web/Desktop/Mobile UI、市场、模板、文档和 Cloud 免费试用都强；自托管成本中等。社区版使用 [LobeHub Community License](https://github.com/lobehub/lobehub/blob/v2.2.10/LICENSE)，商业派生分发需注意许可。“7×24”主要依赖定时任务，不是无限自治；“Continual Learning”是 Memory 提取；“自动组队”仍需成员/模板与 Moderator；默认 canary 页面不能代表稳定版。

**为何今天关注：** 事实是 v2.2.10 集中新模型、worktree、Codex/Claude Code、Workspace 与自动运行可靠性改进，今天 canary 又高频发布。推断上，“很多 Agent，但缺统一操作台”的痛点、产品重新定位和版本曝光共同推动上榜。

## 8. DeepTutor

**一句话定位：** 面向自主学习者、研究者、教师和内部知识团队的开源自托管 AI 学习工作台，把资料问答、解题、测验、研究、可视化、掌握度练习、写作和长期个性化集中到可持续复用的学习空间。

**典型场景：** 上传教材做引用问答；弱项测验与间隔复习；把知识库编译成活书；在 IM 中运行长期 Partner；通过 CLI/API/Skill/Subagent 自动化学习研究。

### 最值得关注的 Features

1. **多模式连续工作区** → Chat、Solve、Quiz、Research、Visualize、Mastery 共用资料和历史 → 避免知识散落 → 连续学习。**状态：稳定；需 LLM，Web 研究需搜索。** [Chat](https://docs.deeptutor.info/explore/chat-workspace/)
2. **Solve/Quiz/Research/Visualize** → 分步解题、练习校验、Web/arXiv/RAG 报告和 SVG/Mermaid/Chart/HTML/Manim → 把答案变成可核验/可练产物 → 数理、综述、教学。**状态：稳定；质量依赖模型和检索。**
3. **多引擎版本化知识库** → LlamaIndex、PageIndex、GraphRAG、LightRAG、外部 Server、Obsidian，并保留旧索引版本 → 按材料和部署选检索 → 教材/论文/团队资料。**状态：默认引擎稳定；高级引擎需 extras/API。**
4. **失败文档单文件恢复** → 即使 KB 为 error 也能删除坏文档后重索引 → 避免整库重建 → 批量 PDF 导入。**状态：v1.5.1 新增；只读/legacy 库有边界。** [v1.5.1](https://github.com/HKUDS/DeepTutor/releases/tag/v1.5.1)
5. **Mastery Path + 题库** → 硬性掌握门槛、错题和间隔复习 → 从聊天完成感转向持续学习 → 备考、技能树。**状态：稳定；需结构化输出可靠的模型。**
6. **Interactive Book** → 从知识库/笔记/题库/聊天生成可审阅大纲与文本、测验、闪卡、代码、动画、交互块 → 把散乱材料变活教材 → 自制课程。**状态：稳定；编译成本/质量依赖 LLM。**
7. **三层 Memory + Graph** → 查看原始事件、场景事实和跨场景画像，并追溯证据、更新、审计、去重 → 个性化可见可修正 → 长期导师。**状态：稳定；模型提炼仍需用户校核。** [Memory](https://docs.deeptutor.info/explore/memory/)
8. **Co-Writer Diff** → 只精修选中文本并在接受前显示完整 Diff → 避免整篇重写 → 笔记、教程、研究报告。**状态：稳定，富编辑体验主要在 Web。**
9. **Partners + 多 IM** → 角色带独立模型、知识、Skills、Notebook、记忆，并接微信、Slack、飞书、Telegram、Teams 等 → 把导师放进日常入口 → 个人/团队辅导。**状态：稳定；需渠道 extras/凭据/allowlist，平台能力不完全一致。**
10. **实时 Subagents/历史导入** → 咨询本机 Claude Code、Codex 或 Partner，或导入其旧会话 → 复用仓库知识和既有调查 → 源码学习、交叉核验。**状态：稳定；需本机 Agent 并明确目录权限。**
11. **Skills/MCP/CLI/API** → `SKILL.md`、EduHub、MCP、NDJSON CLI 与 HTTP/WebSocket/OpenAPI → 既供人使用也可作为自动化后端 → 批处理研究、自定义前端。**状态：稳定；外部服务和多用户授权需配置。** [CLI/API](https://docs.deeptutor.info/cli/)

**同类与缺口：** 相比 NotebookLM，强在自托管、多引擎、Mastery、活书、Memory、IM、CLI/API，弱在零部署和统一媒体体验；相比 Open WebUI/AnythingLLM，教学闭环更强；相比 Moodle，缺课程注册、作业/成绩册、SCORM/LTI 与机构班级分析。

**易用性与限制：** Web UI、PyPI/Docker/CLI-only、OpenAPI、Persona、Office Skills 和完整文档齐全，但全功能需 Python 3.11+、Node 20+、约 4GB 内存、LLM 和 Embedding；解析器/渠道可引入额外依赖和费用。“33 Providers”只是连接覆盖，不保证工具/结构化输出等同；“有引用”不保证结论正确；Memory 画像不是客观事实。聊天附件完整多模态抽取、Immersive Reading 等仍在 [Roadmap](https://github.com/HKUDS/DeepTutor/issues/498)。

**为何今天关注：** 事实是 v1.5.0 汇总 Partners、多知识引擎、Guided Learning 和三层 Memory，v1.5.1 又解决 error KB 单文件恢复。推断上，“开源 NotebookLM + Tutor + Agent/IM 平台”的组合及活书/Codex 会诊提供了强演示点。

## 9. mattpocock/skills

**一句话定位：** 面向真实软件工程团队的可组合 Agent Skills，把需求澄清、规格、任务拆分、TDD、调试、评审、架构维护和跨会话交接固化为重复工作流，同时让人保留控制权。

**典型场景：** 需求访谈和术语固化；Spec/Tickets；TDD 与诊断；双轴 Review；大型项目决策地图与跨会话 Handoff。

## 10. GitHub Copilot SDK

**一句话定位：** 为应用与平台团队把 Copilot CLI 背后的编码 Agent——规划、工具调用、文件编辑、多轮会话和流式事件——嵌入自己的产品，省去从零搭建 Agent 循环。

**版本口径：** 项目整体已 GA；六种官方语言为 Node/TS、Python、Go、.NET、Java、Rust，本次稳定轨均为 1.0.7。但 GA 不代表每项能力在六语言完全同日对齐，Fleet/Agent 管理/FFI 等仍实验。

**典型场景：** IDE/内部 Web 工具嵌入编码助手；CI 故障诊断/迁移；多租户企业 Agent；可恢复长任务；BYOK/MCP 接企业模型与工具。

### 最值得关注的 Features

1. **内置 Coding Agent Loop** → 直接获得规划、工具、文件编辑和多轮会话 → 从模型调用快速进入可执行 Agent → 编码助手、维护机器人。**状态：GA，认证后可用。** [GA](https://github.blog/changelog/2026-06-02-copilot-sdk-is-now-generally-available/)
2. **Streaming/Steering/Queueing** → 订阅 40+ 事件、运行中改变方向或排队后续任务 → 构建有进度、可纠偏体验 → 长重构、Web Agent。**状态：核心可用；语言示例覆盖不完全一致。**
3. **Custom Tools/MCP/Tool Search** → 注册应用函数、连接 stdio/HTTP MCP，并按需发现工具 → 接企业 API且减少 Schema 占上下文 → 内部平台。**状态：Tools GA；MCP 文档称 evolving；Tool Search 进入 1.0.7。** [v1.0.7](https://github.com/github/copilot-sdk/releases/tag/v1.0.7)
4. **Custom Agents/Skills/Fleet** → 配角色、工具和 Skill，由主 Agent 委派或并行 → 降低上下文污染 → 研究/审查/测试分工。**状态：Custom Agents/Skills 可用；Fleet/管理 RPC 实验且 Java 示例缺失。**
5. **Hooks/权限/System Prompt 分段** → 在输入、工具、会话、错误节点审批、改写、脱敏和审计 → 实现企业安全策略 → 高风险工具。**状态：GA 能力，但策略由宿主实现；`approveAll` 不是生产默认。**
6. **图像输入** → 发送截图/图表并接收图像工具结果 → 构建视觉调试助手 → UI/图表分析。**状态：需 Vision 模型；SVG 不支持，多图/大图受限。**
7. **持久/Infinite Sessions** → 保存对话、计划和工作区并自动压缩 → 进程重启后继续长任务 → 迁移/持续修复。**状态：可用；BYOK Key和内存工具状态不持久，无内置锁。**
8. **Cloud/Remote Sessions** → 在 GitHub 托管环境执行，或用 Mission Control Web/Mobile 查看接管 → Agent 脱离当前终端持续工作 → 后台任务、移动审批。**状态：需 entitlement、认证、组织策略和 GitHub 仓库。**
9. **Copilot Auth + BYOK/BYOM** → 用 GitHub 身份/订阅，或接 OpenAI、Azure、Anthropic、Ollama、vLLM/LiteLLM → 同一 SDK 覆盖个人、SaaS、本地 → 多部署。**状态：GA；标准模式消耗 Copilot，BYOK由 Provider 计费，Entra/Managed Identity需宿主桥接。**
10. **多租户部署模式** → 独立/共享 Runtime、每会话 Token、`empty` 安全模式、工具白名单和清理 → 在服务中承载多用户 → SaaS。**状态：提供模式，不是自动隔离平台；应用仍负责 Auth、锁、存储和文件系统边界。**
11. **OTel + Session Budget** → 输出 OTLP Trace，并给会话设置 AI Credits 软上限 → 诊断延迟、工具故障和成本 → 生产监控。**状态：可用；预算可能被最后一次调用越过。**
12. **六语言交付** → 在主流后端栈使用相近 API → 减少重复封装 → 多语言企业。**状态：GA 1.0.7；Node/Python/.NET 可解析/下载固定 Runtime，Go/Java/Rust 默认还需 CLI/打包，Java不支持打包。**

**同类与缺口：** 相比 LangGraph/AutoGen/Semantic Kernel，优势是现成 coding runtime、GitHub 身份和 Mission Control，弱在任意 DAG/状态机；相比模型厂商 SDK，六语言和 GitHub 集成更强，但 Provider-native 控制未必全透传；相比终端 Agent，它能嵌入产品，却不自动获得完整 CLI UI。

**易用性与限制：** 有六语言包、Getting Started、Cookbook、示例、MCP/OAuth/OTel 文档；无内置通用 GUI。默认 CLI-like 模式工具面较广，多租户必须从 `empty` 白名单开始。会话持久化不等于 durable workflow；BYOK 也不消除模型兼容/限流差异。“六语言”不等于 Feature 完全对齐。[兼容性表](https://docs.github.com/en/copilot/how-tos/copilot-sdk/troubleshooting/compatibility)

**为何今天关注：** 事实是 SDK 6 月 GA，昨天发布 1.0.7，新增 Tool Search、工具来源/metadata、企业设置传递和实验 FFI。推断上，稳定版发布、六语言、BYOK/MCP、MIT 与官方品牌共同触发了今日关注，Tool Search 也正面回应大规模工具上下文膨胀。

## 11. UI Skills

**一句话定位：** 面向设计工程师和 Codex、Claude Code、Cursor 等编码 Agent 的 UI Skill 导航与聚合目录，帮助 Agent 在动手前选择少量针对性设计/审查指导；它不是 UI 生成器或自动检查器。

**版本口径：** npm latest 为 [v0.2.3](https://github.com/ibelick/ui-skills/releases/tag/v0.2.3)：110 条、24 分类、5 个自带 Skill；当前 main/官网已是 179 条、26 分类、6 个自带 Skill，但领先稳定包约 30 commits，不能混写。

**典型场景：** 为 UI 任务选 1–3 个最相关 Skill；审计视觉层级/间距；专项修可访问性、Metadata、Motion；按框架找规则；通过外部 installer 持久安装。

### 最值得关注的 Features

1. **Root Router** → `npx ui-skills start` 输出选择协议，通常只选 1 个、复杂任务最多 3 个 → 避免整库塞进上下文和规则冲突 → UI 任务起步。**状态：v0.2.3 稳定、可离线打印。**
2. **分类浏览** → `categories` 输出 accessibility、motion、visual、framework 等主题 → 先缩小问题域 → 不知选什么时。**状态：v0.2.3 为 24 类；main 26 类未发布。**
3. **列表/过滤** → `list` 或 `list --category motion` 展示命名空间、主题和简介 → 快速比较候选 → Skill 选型。**状态：稳定；来自打包 Registry，不会自动同步 main。**
4. **完整 Skill 获取** → `get <slug>` 输出自带或远端 `SKILL.md` → 直接交给 Agent → 专项执行。**状态：稳定；第三方需联网且跟随浮动分支。**
5. **命名空间消歧** → 同名 Skill 要用 `owner/slug` 精确选择 → 避免取错作者版本 → 多来源目录。**状态：稳定，有明确候选/错误提示。**
6. **Web 可视目录** → 搜索、按 Topic 浏览、读全文、复制命令和相邻推荐 → 人可先审阅再加载 → 人机协作选型。**状态：官网跟随 main，不是 v0.2.3 冻结界面。** [官网](https://www.ui-skills.com/)
7. **机器可读 Registry/llms.txt** → 暴露条目、URL、简介与站点索引 → 便于 Agent/脚本发现 → 自动化检索。**状态：线上可用；无版本化 JSON Schema/锁文件承诺。** [Registry](https://www.ui-skills.com/skills/registry.txt)
8. **多源聚合** → 一处发现 Anthropic、Vercel、Microsoft、Vue、Remotion、个人作者等内容 → 节省跨仓库查找 → 前端/动效/测试。**状态：官网 179 条；数量是索引，不是同标准认证数。**
9. **自带专项 Skills** → baseline-ui、fixing-accessibility、fixing-metadata、fixing-motion-performance 与 root router → 覆盖高频 UI 修复 → 日常前端。**状态：v0.2.3 的 5 个稳定；规则对 Tailwind/React 有偏好。**
10. **`improve-ui` 审计计划** → 只审计、不改产品源码，基于现有设计证据生成交接计划 → 降低无依据重设计 → 多 Agent 审计/实施。**状态：main/官网已上线，未进入 npm v0.2.3。**
11. **跨 Agent 安装入口** → 页面给出 `npx skills add ... --skill ...` → 把选中内容装入 Codex/Claude/Cursor 等 → 项目级复用。**状态：依赖外部 `vercel-labs/skills` CLI；`ui-skills` 本身没有 add/update/remove。**

**同类与缺口：** 相比 skills.sh/installer，它更聚焦 UI 分类与路由，但不负责安装生命周期；相比 Hallmark，它广而可组合，却没有统一端到端设计方法、项目记忆和门禁；相比单个 frontend-design Skill，它的独有价值是发现与最小上下文选择。

**易用性与限制：** Web、CLI、纯文本 Registry、llms.txt 和复制安装命令齐全，无账号/Key；但“179 个 high-quality”没有公开评分/Benchmark，评分和 before/after 仍是 Issue。178 个 GitHub 第三方链接几乎都跟随 main/master，无 commit pin、签名、内容哈希、安全扫描或统一许可展示；仓库 MIT 不自动覆盖第三方内容。`get` 只是打印 Markdown，真正修复仍由宿主 Agent 完成。

**为何今天关注：** 事实是 7 月 13–16 日连续增加 Skill、`improve-ui` 和 Agent 专页，官网已扩到 179 条。推断上，Skills 爆发后“该选哪个、如何少加载”成为新痛点，UI 同质化焦虑与低门槛 `npx` 入口共同带来热度。

## 12. Graphify

**一句话定位：** 面向使用 AI 编程助手的开发者与团队，把代码、文档和媒体整理为可持久查询、可共享、带证据标签的项目知识图，减少 Agent 每次会话重新 grep、重读文件的成本。

**典型场景：** 接手陌生仓库；跨会话复用上下文；查询调用/依赖路径和 PR 影响；连接 ADR/PDF/截图/会议音频；通过 MCP/HTTP 与团队共享。

### 最值得关注的 Features

1. **本地代码地图** → 抽取函数、类、调用、导入、继承和跨文件关系，代码-only 无需 LLM/API → 私有源码不必上传且快速得整体结构 → Onboarding、Agent Context。**状态：基础开箱；不同语言深度不等价，部分需 extras。**
2. **多模态统一图** → 把 Markdown、PDF、Office、图片、音视频和 Google Workspace 与代码连接 → 同时回答“做什么”和“为什么” → 研究库、设计文档。**状态：需对应 extras；文档/图片语义 Pass 会用 IDE 模型或自配后端。**
3. **证据/置信度标签** → 每条边标 `EXTRACTED`、`INFERRED`、`AMBIGUOUS` 并留来源 → 用户可优先审查不确定关系 → Agent 引用、影响分析。**状态：开箱；标签不等于推断必然正确。** [How it works](https://github.com/Graphify-Labs/graphify/blob/v0.9.17/docs/how-it-works.md)
4. **`query/path/explain`** → 取相关子图、查最短路径、解释节点和连接 → 把整库压成问题相关关系 → 数据链、符号职责。**状态：开箱；擅长结构关系，不是通用模糊语义问答。**
5. **交互地图与报告** → 生成 `graph.html`、`GRAPH_REPORT.md`、`graph.json`，展示关键节点、社区和设计理由 → 人和 Agent 都能消费 → 架构盘点。**状态：开箱；静态 Web 图不是完整 IDE/协作控制台。**
6. **20+ Agent 安装目标** → 为 Claude Code、Codex、Cursor、Copilot、Gemini 等写入 Skill/规则/Hook → 同一图跨工具使用 → 多 Agent 团队。**状态：基础可用；平台能力不同，Codex 多 Agent 需显式开启。**
7. **MCP stdio/HTTP** → 暴露 query、neighbor、shortest path、PR 工具，HTTP 可团队共用 → Agent 无需解析文件 → IDE/团队共享。**状态：需 `mcp` extra；外部监听应设 API Key，无托管服务。**
8. **增量/Watch/Git 协作** → 哈希跳过未变文件，Hook 在提交/切分支更新，merge driver 合并团队图 → 降低长期维护成本 → 活跃仓库。**状态：可用但仍在硬化；v0.9.17 修复语义层丢失、重复节点和 merge 问题。**
9. **多种导出** → Wiki、Obsidian、GraphML、SVG、call-flow、Neo4j、FalkorDB → 进入现有知识/图工具 → 团队分享。**状态：部分需 extras/外部服务。**
10. **跨项目与 PR 工作台** → 注册多个项目，查看 PR 的 CI、Review、worktree、图影响和冲突，模型可 Triage → 从单仓扩到审查队列 → Monorepo/多服务。**状态：基础 CLI 已有；AI 排序需模型，不等同 Enterprise merge gate。**
11. **工作记忆/反思** → `save-result` 记录 useful/dead_end/corrected，`reflect` 汇总 `LESSONS.md` 并提示过期经验 → 除项目结构外积累团队实践 → 重复排障。**状态：CLI 已提供但较新，无完善管理 UI。**

**同类与缺口：** 相比 Aider Repo Map，Graphify 强在持久图、路径、证据、多模态、MCP 和跨助手，弱在没有内置代码编辑闭环；相比 Sourcegraph/Cody，强在本地、MIT、免账号和开放制品，弱在企业权限/多仓/IDE 体验；相比 GraphRAG，代码结构更直接，但大规模散文的模糊语义检索更弱。

**易用性与限制：** `uv tool install graphifyy` + `graphify install` 两步注册，提供 CLI、Slash Skill、MCP、静态 UI、JSON/GraphML/图数据库出口和 worked 示例。注意 PyPI 包名为双 y、Python 3.10+、extras 较多。“完全本地”只严格适用于代码图与本地图存储；语义 Pass 默认可能发给 IDE/云模型，需 Ollama 才全本地。52 文件样本的 71.5× Token Reduction 不能泛化；项目方 Benchmark 样本小且部分指标不领先 Dense RAG；16K 文件 Monorepo、超长语义 Chunk、同名文件碰撞等仍有开放 Issue。[Benchmark](https://github.com/Graphify-Labs/graphify/blob/v0.9.17/BENCHMARKS.md)

**状态边界：** v0.9.17 为 Pre-1.0；`--mode deep` 的 Skill 侧完整传递仍待 follow-up。Enterprise 的 merge-gate、graph-aware review、Digest、Jira/Atlassian Connector 仅 Early Access，不能计入开源 GA。[Enterprise](https://www.graphify.com/enterprise)

**为何今天关注：** 事实是昨日 [v0.9.17](https://github.com/Graphify-Labs/graphify/releases/tag/v0.9.17) 带来查询加速和大量增量、缓存、隐私、解析修复。推断上，“query instead of grep”、local-first、MCP、多 Agent 和可视图都高度适合传播；高频修复也说明仍在快速收敛。

## 今天最值得体验的 3 个项目

1. **[Open Interpreter](https://github.com/openinterpreter/openinterpreter)**：今天的 v0.0.28 与 Kimi K3 发布直接相关，最适合验证“同一 Codex 风格入口 + 不同模型 Harness”能否降低 Coding Agent 成本。先用 `workspace-write/on-request`，不要用 `--yolo`。
2. **[Bonsai-demo](https://github.com/PrismML-Eng/Bonsai-demo)**：最适合验证 27B 低比特本地模型是否已进入个人硬件可用区间。优先体验 Vision、Reasoning Effort、Tool Calling 和 MCP，不要先追求 262K 全窗口或手机部署。
3. **[DeepTutor](https://github.com/HKUDS/DeepTutor)**：今日最完整的垂直 Feature 链，从知识库、Research、Mastery、Memory 到 Interactive Book、Partner 和 Codex/Claude 会诊，适合判断长期 Agent 工作区是否比一次性问答更有价值。

> 若你的问题是“AI 生成页面太模板化”，Hallmark 是最低成本的替代体验；若问题是大型仓库上下文，Graphify 应替换 DeepTutor 进入 Top 3。

## 值得持续跟踪的新 Feature 方向

1. **Agent Operating Layer**：Groups、任务状态、排程、验收、运行报告、消息渠道和人工介入会从聊天附属功能发展为独立操作层。
2. **Harness 与 Runtime 可移植性**：一边是开放模型切换不同 Harness，一边是六语言 SDK 嵌入成熟 Runtime；竞争焦点会从模型 API 转向完整 Agent 行为。
3. **生产上下文闭环**：Trace 本身已不稀缺，下一阶段是生产样本如何进入 Eval、Prompt/模型实验、用户行为指标和修复 PR。
4. **可审计记忆与关系上下文**：DeepTutor 的证据 Memory、LobeHub 的白盒规则、Graphify 的证据边都说明用户既要“记住”，也要知道来源和可删除性。
5. **Agent 的业务语义标准**：Ossie 代表把指标、同义词、说明和样例问题标准化；真正价值取决于厂商原生采用和查询 Runtime，而非只发布 Schema。
6. **本地多模态 Tool Model**：低比特模型开始同时交付 Vision、Thinking、Tool Calling、MCP 和 OpenAI API；后续要看独立质量复现、长链可靠性和主线运行时兼容。
7. **Skill 发现、评分与供应链治理**：Skills 从“有没有”进入“选哪个、能否信任、是否冲突、如何固定版本”；UI Skills 的路由只是早期答案。
8. **设计 Agent 的视觉反馈闭环**：Hallmark 已把结构和审计规则化，下一步是截图、浏览器交互、视觉回归和人类反馈形成可重复 Eval。
