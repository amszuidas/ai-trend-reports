# Feature 深度卡片

## 1. Apache Ossie

**一句话定位：** 为数据平台、BI/AI 工具厂商和数据团队提供厂商中立的语义模型交换规范与参考转换工具，减少指标、字段和业务定义在不同系统间漂移；它不是可直接查询数据的语义层平台或 Agent 产品。

**典型场景：** 在 dbt、GoodData、Omni、Salesforce 等语义格式间迁移；以 Git 统一维护指标口径；给数据 Agent 提供结构化业务语义；在 CI 校验模型；为新平台开发兼容转换器。

### 最值得关注的 Features

1. **YAML/JSON 核心规范** → 描述 semantic model、dataset、field、metric 和 relationship → 让业务语义脱离单一厂商并可版本控制 → 跨团队共享与迁移。
2. **统一指标与关系表达** → 支持顶层指标、主键、唯一键及复合关系 → 保留基本业务口径和联表语义 → 销售、客户、财务模型。
3. **多 SQL 方言表达式** → 同一字段可携带 ANSI、Snowflake、Databricks、BigQuery、MDX、Tableau、MAQL 等表达 → 减少跨平台重新翻译 → 多仓库、多 BI 环境。
4. **分层 AI Context** → 模型、数据集、字段、关系、指标均可附 instructions、synonyms、examples → 为 LLM 补充词义和使用约束 → Text-to-SQL、指标问答。它只定义上下文，不运行 Agent。
5. **Vendor Custom Extensions** → 保存目标平台无法映射的厂商专属 JSON → 提高导入后再导回的保真度 → 渐进迁移；其他厂商通常不能消费这些字段。
6. **参考转换器集合** → 已覆盖 dbt、GoodData、Honeydew、Omni、OrionBelt、Polaris、Salesforce 等双向转换，Snowflake 为单向 → 可真实验证格式交换 → 数据栈迁移。
7. **dbt MetricFlow 互操作** → 在 `semantic_manifest.json` 与 Ossie YAML 间转换并报告丢失能力 → 接入现有 dbt 项目 → 指标层试验。
8. **Schema + Validator** → 检查 JSON Schema、重复名称、关系引用和部分 SQL 方言语法 → 把低成本错误前移到提交/CI → 模型治理。
9. **Python 类型模型** → 用 Pydantic 读写 YAML/JSON → 转换器和内部工具无需手写解析 → 平台集成。
10. **Ontology 与表达式草案** → 提供概念层 schema、Flights 示例和便携表达式提案 → 展示未来统一语义方向 → 标准研究；表达式文档仍是提案，不是执行引擎。

**可用状态与成本：** 核心规范、Schema、TPC-DS/Flights 示例和验证器可直接从源码使用；Polaris/Salesforce/dbt 等转换器需要相应凭据、JDK/Maven 或预生成 manifest。项目无 GitHub 正式 Release，仅见 `osi-0.1.1-rc1` tag，ASF 也尚未列出正式 Release；统一 CLI、Registry、Sync API、语义查询语言、SQL 执行、权限、verified queries 和 AI 暴露控制仍在 Roadmap。[核心规范](https://github.com/apache/ossie/blob/main/core-spec/spec.md) · [转换器](https://github.com/apache/ossie/tree/main/converters) · [验证器](https://github.com/apache/ossie/blob/main/validation/validate.py) · [Roadmap](https://github.com/apache/ossie/blob/main/ROADMAP.md)

**同类比较：** 相比 dbt MetricFlow 和 Cube，Ossie 的独特强项是厂商中立格式、多方言表达和转换生态；MetricFlow 更擅长实体/粒度/复杂指标与 SQL 生成，Cube 已有查询 API、权限、缓存和预聚合运行时。Ossie 缺少 UI、托管服务、查询 API、权限执行和端到端在线 Demo，更像“语义层的 OpenAPI”而不是运行平台。

**限制和营销边界：** “Write once, query anywhere”目前只是尽量转换；多种转换会丢失或 stash-only 保存字段；所谓 lossless 多指同一厂商格式往返。Validator 不执行 SQL，也不保证关系列真实存在或业务结果正确。“AI-ready grounding”是用途，不等于内置 LLM、MCP、Text-to-SQL 或准确率保证。

**为何今天关注：** 事实是项目 2026 年 6 月进入 Apache Incubator，近期完成 OSI→Ossie 更名并密集增加转换器/表达式文档。推断上，Apache 治理、dbt 采用信号和“给 Agent 统一业务语义”的叙事共同推动了热度；真正观察点是能否形成稳定 Release 和厂商原生支持。

## 2. Hallmark

**一句话定位：** 为 Claude Code、Cursor、Codex 用户提供“反 AI 模板感”的前端设计 Skill，通过结构轮换、主题规则、审计和质量门禁，让页面更像为具体品牌设计，而不是同一 SaaS 模板换色。

**典型场景：** 从零构建品牌页/落地页；审计现有页面；保留文案与信息架构进行重设计；从截图/URL 提取设计 DNA；为多页面项目固化设计 Tokens。

### 最值得关注的 Features

1. **Build 默认流程** → 先读取 brief 和现有字体、颜色、框架、动效依赖，再选择结构与主题 → 减少破坏已有品牌/CSS → 新页面或增量开发。
2. **Audit** → 输出按 critical/major/minor 排序的问题位置、反模式和修正建议，明确不改文件 → 形成可复核 Punch List → 上线前设计 Review。
3. **Redesign** → 保留路由、品牌、文案意图和信息架构，重做节奏、标题、组件语气 → 比无约束重写更适合真实仓库 → 旧页面翻新。
4. **Study** → 从截图或公开 URL 提取结构、组件、字体角色、颜色锚点和节奏，并可导出 `design.md` → 借鉴设计语言而非像素克隆 → 参考研究。
5. **结构与组件轮换** → 内置 21 种 macrostructure、14 种导航、8 种页脚及约 50 个组件 archetype，近期方案不重复 → 真正改变页面骨架 → 连续生成多页面。
6. **20 主题 + Custom** → 主题目录覆盖 editorial、minimal、atmospheric、playful；品牌意图不匹配时走 Custom，生成一次性色板/字体/结构 → 兼顾速度与差异化 → 品牌项目。
7. **生成前/后双门禁** → 六轴 pre-emit critique 后再跑字体、渐变、交互、对比度、响应式、虚构数据等 Slop Test → 把审美建议变为检查清单 → 交付前自检。
8. **项目感知与跨运行记忆** → `.hallmark/preflight.json` 缓存扫描，`.hallmark/log.json` 保存最近 20 次结构、主题和素材策略 → 主动避开近期方案 → 长期项目。
9. **组件八状态** → 单组件任务要求 default、hover、focus、active、disabled、loading、error、success，并生成预览 → 比静态截图更接近产品级组件 → Button/Input/Modal。
10. **设计系统可移植** → 构建时输出 `tokens.css`，用户明确锁定 DNA 后才生成 `design.md`，可扩展 Tailwind/DTCG/shadcn 变量 → 让其他 Agent 和页面复用 → 多页面交接。

**可用状态与成本：** 四个 verbs、20 主题、Custom、门禁、项目记忆和示例可通过 `npx skills add nutlope/hallmark` 安装，MIT 免费；模型、付费字体和外部素材可能收费。它没有独立编辑器、SDK 或业务 API，`hallmark audit/study` 是 Agent 指令而非传统 CLI。URL Study 依赖宿主网页读取能力，认证页/JS 空壳页通常需要截图。[SKILL](https://github.com/Nutlope/hallmark/blob/main/skills/hallmark/SKILL.md) · [Live Demo](https://www.usehallmark.com/) · [Recipes](https://github.com/Nutlope/hallmark/blob/main/docs/recipes.md)

**同类比较：** 相比 Anthropic frontend-design Skill，Hallmark 的强项是四种任务模式、明确结构/主题目录、跨运行日志、组件八状态、可审计门禁和 `design.md` 交接；代价是规则更重、自由度更低。缺少自动截图—视觉反馈闭环、真正的相似度检测、图表专项和一键 variant。

**限制和营销边界：** README 写 57 gates，当前规则实际为 1–57 加 38a，共 58 项，数字已漂移。Custom 已可用；Nanobanana、Brand-first、`variant`、图表、代码库 Study、Explain、MCP Live Preview 仍属 [Roadmap](https://github.com/Nutlope/hallmark/blob/main/ROADMAP.md)。仓库无正式 Release/Tag，Recipes 与当前规则也有局部不同步。“拒绝 AI slop”是设计约束，不是客观审美 Benchmark。

**为何今天关注：** 事实是榜单第 2、单日 +1,277，且一行安装、互动 Demo 和成品对比让价值非常直观。推断上，“AI 页面千篇一律”是高频痛点，Anti-AI-slop 定位、Audit/Study 和持续项目记忆共同带来传播力。

## 3. PostHog（边界纳入：AI 功能子集）

**一句话定位：** 面向 AI 产品工程团队，把 LLM Trace、质量评估、提示词实验和真实用户行为数据放在同一平台；PostHog 本身仍是综合产品数据平台，而非纯 AI 项目。

**典型场景：** 调试 RAG/客服/Agent 调用链；监控成本与延迟；用自动和人工 Eval 检查幻觉、安全和相关性；从生产 Trace 建立回归数据集；比较模型或 Prompt 版本；让 Agent 读写产品数据。

### 最值得关注的 Features

1. **LLM Traces/Generations/Spans/Sessions** → 记录输入输出、Token、模型、延迟、成本、错误和 Trace ID → 还原一次多步对话 → Agent、RAG、工具调用。
2. **AI 使用与成本分析** → 按模型、用户、组织、会话拆解成本、延迟、错误 → 定位昂贵功能和回归 → FinOps、上线监控。
3. **产品数据联动** → `$ai_*` 事件可进入趋势、漏斗、SQL、告警并关联 Replay/Error Tracking → 看见失败影响到哪些真实用户 → AI Feature 产品化。
4. **在线 AI Evals** → 支持 LLM Judge、Hog 规则和本地情感模型，可采样与过滤 → 持续检查帮助性、幻觉、越狱、毒性和格式 → 生产质量门禁。
5. **人工 Trace Reviews** → 通过队列、Scorer 和备注批量审查真实 Trace → 把事故、差评和支持工单转成结构化 QA → 高价值会话复盘。
6. **Playground** → 并排比较模型、Prompt、参数、工具调用，并从生产 Trace 重放 → 无需部署即可复现失败 → Prompt 调优、模型选型。
7. **Prompt Management（Beta）** → 管理模板变量、不可变版本、Diff、恢复、SDK 缓存与 fallback → 让 Prompt 变更可追溯 → PromptOps。
8. **Prompt A/B Experiments** → 用 Feature Flag 分流多个 Prompt，以成本、延迟和 Eval 通过率衡量 → 用真实流量验证改动 → 渐进发布。
9. **聚类、摘要与情感** → 聚类相似 Trace、标记离群并生成流程摘要/用户情绪 → 从大量对话中发现系统性问题 → 运营洞察。
10. **Datasets/离线回归（早期）** → 整理生产输入输出并重放改动 → 上线前发现回归 → Golden set；官方文档仍在编写，不能等同成熟实验套件。
11. **PostHog AI、MCP 与 Wizard** → 自然语言查数据、写 SQL、建 Insight/Flag/Survey，或让 Codex/Claude/Cursor 通过 MCP 操作；Wizard 自动安装埋点 → 降低分析与接入门槛。

## 4. Open Interpreter

**一句话定位：** 为希望用 DeepSeek、Kimi、Qwen、本地模型等低成本模型完成真实软件工程任务的开发者，提供 Codex 式终端体验、多 Harness、原生沙箱和可扩展 Agent 工作流。

> 本卡片只讨论 2026 年 7 月发布、基于 Codex 的新 Rust 版；原 Python 版已迁至社区维护的 [endolith/open-interpreter](https://github.com/endolith/open-interpreter)，其 SDK/Computer API 等旧能力不计入当前项目。

**典型场景：** 本地修 Bug/重构/补测试；切换低价或本地模型；在 CI 中自动 Review/修复；通过 ACP/MCP 嵌入编辑器和内部工具；修改 UI 后执行 Web/桌面 QA。

### 最值得关注的 Features

1. **模型 Harness 仿真** → `/harness` 切换 native、Claude Code、Kimi、Qwen、DeepSeek、SWE-agent 等提示词/工具/message surface → 让特定训练分布的低成本模型使用熟悉接口 → 比较同一模型的 Agent 表现。它不启动真正的外部 CLI。
2. **多 Provider 与本地模型** → `/model` 选择 Provider、模型、推理强度和 Harness，覆盖 OpenAI、Bedrock、Ollama、LM Studio 及大量兼容目录 → 按价格、隐私、地区切换 → 企业网关、本地推理。
3. **完整终端编码 Agent** → 读写文件、运行命令、规划、审查 Diff、附截图、保持后台进程，并提供 `/review`、`/plan` → 从问答进入可验证修改 → 日常开发。
4. **非交互 `exec`** → 支持 stdin、JSONL、JSON Schema、保存最终消息、超时和额外验证回合 → 易接入自动化 → CI Review、批量报告。
5. **原生沙箱与权限** → macOS Seatbelt、Linux/WSL bubblewrap/seccomp、Windows 原生边界，可限制文件、域名、私网和 socket → 缩小执行爆炸半径 → 长任务/陌生仓库。细粒度 profile 仍是较新 Beta。
6. **Computer Use/QA Skill** → Web 使用 `agent-browser`，桌面使用 `cua-driver`，强调操作前后状态对比 → 验证真实界面而非只编译 → E2E QA；驱动需按需联网安装。
7. **ACP 编辑器接入** → `interpreter acp` 通过 stdio 提供会话、流式输出、工具进度和权限请求 → 无需解析终端文本 → Zed/JetBrains；VS Code 依赖社区扩展。
8. **MCP 双向能力** → 既消费本地 stdio/远程 HTTP MCP，也可用 `interpreter mcp-server` 被其他 Agent 调用 → 接入文档、工单、数据库或成为 Agent-of-agents → 企业工作流。
9. **Skills、Hooks、AGENTS.md** → 封装流程、在会话/工具/审批/Subagent 生命周期执行可信脚本，并分层加载仓库规则 → 固化团队规范 → 多项目协作。
10. **本地会话、分叉与 Subagents** → resume、fork、compact，并让独立线程并行调查 → 支持跨天和并行任务 → 大型仓库。

**可用状态与成本：** 安装器、TUI、exec、会话、Review、基础沙箱/审批、AGENTS.md、Hooks 和 Subagents 可用；模型 Key/Ollama、MCP、ACP 客户端、精细权限和 Computer Use 驱动需配置。本体 Apache-2.0，模型/第三方服务另计费。Plugins、Memories、Apps、Undo 默认关闭或实验；远程 app-server 属高级接口。[README](https://github.com/openinterpreter/openinterpreter) · [Harness](https://www.openinterpreter.com/docs/terminal/harness) · [Sandbox](https://www.openinterpreter.com/docs/terminal/sandbox) · [MCP](https://www.openinterpreter.com/docs/terminal/mcp)

**同类比较：** 相对 Codex，最大差异是广泛 Provider 与按模型切换 Harness，但没有 Codex Cloud；相对 Claude Code，模型选择/本地推理更广、OS 沙箱更强，但缺少一方模型与 IDE/云深度整合；相对 Aider，强在沙箱、ACP/MCP、结构化自动化、Hooks、Subagents 和 UI QA，弱在成熟 Git-first 提交/撤销和自动 lint/test 工作流。

**限制和营销边界：** Harness 有 wire API 约束，弱模型接通也未必可靠用工具。沙箱不自动覆盖 MCP、浏览器、远程服务和用户已批准的越权动作；Windows 与 macOS/Linux 约束不同。当前约 6.5 万 Stars 主要继承自 2023 年旧项目，不能解释为新 Rust 版新增用户。

**为何今天关注：** 事实是官方 7 月 13 日发布新 Rust 版，7 月 14–15 日连续发 `0.0.22`–`0.0.25`。推断上，“Codex 基座 + 低成本模型 Harness + 原生沙箱”的新定位和旧社区重新关注，共同带来今日 +299。

## 5. Bonsai-demo

**一句话定位：** 为希望在个人电脑、单卡 GPU 甚至高端手机上运行本地模型的开发者，把超低比特 Bonsai 27B 的聊天、视觉、推理、工具调用、MCP 和 Agent Demo 打包成脚本化体验。

### 最值得关注的 Features

1. **27B 超低内存模型** → 提供 1-bit 与 Ternary 两个取舍版本 → 让 27B 级模型进入普通笔记本/消费级 GPU 的内存预算 → 本地助手、边缘部署。
2. **一键安装与型号切换** → `setup.sh`/`setup.ps1` 处理依赖、模型、运行时和 WebUI，可切 27B/8B/4B/1.7B 与两种家族 → 降低首次运行门槛 → 跨硬件评估。
3. **本地多模态视觉** → 27B GGUF 可在 UI 和 OpenAI 兼容 API 接收图片 → 本地分析照片、截图和图中文字 → 隐私敏感视觉任务。

5. **原生结构化 Tool Calling** → 接收 OpenAI 风格 `tools` 并返回 `tool_calls` → 不必靠提示词解析函数调用 → Agent/自动化集成；工具执行仍由客户端完成。
6. **MCP Agent Loop** → llama-server UI 内置 MCP 客户端和有限轮次工具循环，并预配置 Hugging Face、DeepWiki → 可直接查询模型与仓库资料 → 研究助手。

8. **长上下文** → 27B 标称 262K，并可按内存调整 → 支持长资料和持续会话 → 报告分析；最大上下文会显著增加 KV Cache 内存。
9. **4-bit KV Cache（实验）** → 降低长上下文缓存占用 → 在紧内存设备扩展窗口 → 百 K 级文档；仅 llama.cpp、默认关闭。
10. **推测解码（实验）** → CUDA 下用 drafter 改善代码/推理速度 → 缓解低比特模型交互延迟 → 单用户请求；Mac、多轮缓存和并发并不适用。
11. **多入口与跨平台** → CLI、内置 Web UI、Open WebUI、OpenAI API，覆盖 CPU/Metal/CUDA/Vulkan/ROCm → 方便从试玩过渡到应用集成。

**可用状态与成本：** 27B Hugging Face 仓库本次核验已公开且不 gated，README 中“暂需 `BONSAI_TOKEN`”已过期。一键安装、视觉、Thinking、Tool Calling 与 OpenAI API 可用；网络 MCP、Brave 搜索、stdio bridge 和 Jupyter 需额外配置。27B 才有多模态；当前 1-bit MLX 路径仍是纯文本。1-bit `Q1_0` 已进入主线 llama.cpp，但 MLX 仍需 fork；Ternary 的 Demo 格式、主线 group-64 格式和计划中的 `PQ2_0` 不能混用。[Vision](https://github.com/PrismML-Eng/Bonsai-demo/blob/main/VISION.md) · [Tools/MCP](https://github.com/PrismML-Eng/Bonsai-demo/blob/main/TOOLS.md) · [Open WebUI](https://github.com/PrismML-Eng/Bonsai-demo/blob/main/OPENWEBUI.md) · [KV Cache](https://github.com/PrismML-Eng/Bonsai-demo/blob/main/KV-CACHE.md)

**同类比较：** 相比 Ollama/LM Studio，它不是通用模型市场，而是对 Bonsai 低比特家族的有主见预配置，强项是把视觉、Thinking、工具、MCP 和分析 Demo 一次配好。缺少成熟桌面安装器、通用模型管理和手机成品 App；相比常规 4-bit 27B 节省内存，但要接受部分质量损失、项目方基准和后端 fork。

**限制和营销边界：** “能放进手机内存”不等于仓库提供即装即用手机 App；长上下文上限不等于低成本运行；PDF 宣传不宜等同完整多页 PDF 工作流；模型卡也把长链 Agentic coding、多文件运行—测试—修复列为弱项。项目公布的能力保留率尚非独立复现。

**为何今天关注：** 事实是 7 月 14 日集中公开 27B 与视觉、MCP、Open WebUI、长上下文文档。推断上，“27B 可在笔记本甚至手机预算运行”与一条命令体验本地多模态 Agent 的组合，是今日 +323 Stars 的主要传播驱动。

## 7. LobeHub

## 10. GitHub Copilot SDK

**一句话定位：** 为希望把成熟编码 Agent 嵌入自有应用、服务或内部平台的开发者，提供六语言 SDK，省去自行实现规划、工具循环、文件操作、会话和权限基础设施。

**典型场景：** 在 IDE/DevOps 门户嵌入编码 Agent；后台自动修复/升级/补测试；配置企业专用 Agent/Skills/MCP；制作带流式进度和人工审批的助手；用共享 Runtime 或 BYOK 服务内部用户。

### 最值得关注的 Features

1. **完整 Agent Loop** → Copilot Runtime 处理多轮规划、模型调用、工具执行和回传 → 应用只需发任务、消费事件 → Bug 修复、重构、自动维护。
2. **六语言 SDK** → Node/TypeScript、Python、Go、.NET、Java、Rust → 企业直接使用现有技术栈 → 内部平台；各语言新 Feature/示例并非永远完全齐平。
3. **流式事件与 Steering** → 增量输出、工具事件、`sendAndWait`、中止、追加/排队指令 → UI 实时展示并干预长任务 → 交互控制台。
4. **自定义 Tools** → 用 JSON Schema/语言类型注册宿主进程工具 → 把数据库、工单、CI 和内部 API 交给 Agent → 企业自动化；可信工具放行也把安全责任交给应用。
5. **Custom Agents** → 为专家配置提示词、工具、MCP、Skills，由主 Agent 委派隔离子任务 → 安全、测试、文档分工 → 多 Agent 应用。

7. **MCP** → 连接 stdio、HTTP/SSE Server，并限制每个 Server 的工具 → 接浏览器、数据库和内部系统 → 工具生态。
8. **Hooks 与权限** → 在工具、Prompt、会话、错误等事件修改参数、拦截或审计 → 构建安全策略 → 生产应用。默认工具面较广，但无 Permission Handler 时并非自动全放行。
9. **持久会话** → 创建、恢复、列举、删除、压缩会话 → 跨进程继续任务 → 长期工作；用户归属和存储隔离仍由宿主负责。
10. **Plugin Directory + OpenTelemetry** → 一次打包 Skills、Hooks、MCP、Agents、LSP，并关联 SDK/CLI Trace → 企业分发与诊断 → 平台化部署。

**可用状态与成本：** Node 和 .NET 捆绑 CLI，Python 首次运行下载 Runtime，Go 可外部安装或构建时嵌入，Java 需 PATH/显式路径，Rust 默认构建时嵌入；“无需安装”不等于没有原生二进制兼容成本。标准认证需要 Copilot 订阅/额度；BYOK 可不用订阅，接 OpenAI、Anthropic、Azure/OpenAI-compatible、Ollama 等并由 Provider 计费。项目声明 GA，稳定版为 [v1.0.6](https://github.com/github/copilot-sdk/releases/tag/v1.0.6)；Fleet、部分 transport 和 `v1.0.7-preview.3` 的 Tool Search 仍属 Preview。[Getting Started](https://github.com/github/copilot-sdk/blob/main/docs/getting-started.md) · [BYOK](https://github.com/github/copilot-sdk/blob/main/docs/auth/byok.md) · [Bundled CLI](https://github.com/github/copilot-sdk/blob/main/docs/setup/bundled-cli.md)

**同类比较：** 相比 OpenAI Agents SDK，强在现成编码 Agent Runtime、文件/终端工具、Skills/插件和六语言 Binding；弱在通用 Handoff、Guardrails、Tracing/Evals、Realtime/Voice 的一等抽象。相比 Claude Agent SDK，语言覆盖更广并融合 GitHub 身份；两者都依赖自家 CLI Runtime。自行编排拥有更高模型/沙箱/流程自由，但要承担恢复、状态、权限、MCP、并发和审计工程。

**限制和营销边界：** 共享多租户服务不应直接使用暴露宿主文件系统的默认模式，应从 `empty` 模式逐项注册工具和工作区。SDK 只暴露 Runtime 的协议能力，CLI 的 TUI、Slash Commands、交互插件管理和模型选择器不能自动算 SDK Feature。“production-tested”描述 Runtime，不保证所有 Binding、Preview API 和多租户部署均已生产验证。

**为何今天关注：** 事实是项目已 GA、稳定版后连续发布 1.0.7 Preview，但今日仅 +8。推断上，六语言覆盖、GitHub 品牌和“把编码 Agent 直接嵌入产品”的需求带来持续关注，而非单日爆发。

## 11. UI Skills

**一句话定位：** 面向设计工程师和编码 Agent 的 UI Skill 注册表、浏览网站与轻量 CLI 路由器，帮助 Agent 按任务选择 1–3 个最相关 `SKILL.md`，避免把大量互相冲突的规则一次塞进上下文。

**典型场景：** UI 改造前选择最小 Skill 组合；清理间距/层级/字体；专项检查无障碍、动画、SEO；查找 Next.js/Vue/React Native/Three.js/SwiftUI 规则；人工比较多个来源的 Skill。

### 最值得关注的 Features

1. **`npx ui-skills start`** → 输出 `ui-skills-root` 路由 Markdown，要求 Agent 判断 UI 任务、分类并优先加载一个 Skill → 降低无关上下文 → 日常 UI 工作。它不会自己执行路由或修改代码。
2. **分类发现** → `categories` 输出 accessibility、motion、systems、visual、interaction、performance、craft、taste、typography、color、framework 等主题 → 先缩小问题域 → 专项审查。
3. **分类列表** → `list`/`list --category motion` 返回路径、主题和描述 → 快速比较候选 → Skill 选型。
4. **按需读取** → `get <slug>` 输出完整 Markdown，本地 Skill 从 npm 包读取，外部 Skill 从 GitHub Raw 获取 → 直接进入 Agent 上下文 → 即时使用。
5. **最小上下文策略** → 默认 1 个、双重问题 2 个、广泛 Review 最多 3 个，并按主题→技术栈→具体程度排序 → 减少规则冲突和 Token → 大仓库。
6. **多来源 Registry** → 统一作者、仓库、Raw/GitHub URL、描述和 Topics，Anthropic frontend-design 只是其中一项 → 更像 UI Skill 包目录 → 跨生态发现。
7. **自带基础 Skills** → v0.2.3 含 `baseline-ui`、accessibility、metadata、motion-performance 和路由器 → 覆盖高频清理 → 新项目。
8. **网站浏览与搜索** → 推荐、Topic、全文、快捷搜索和复制 → 人可先审阅来源 → 避免盲目加载。
9. **跨 Agent 导航** → main 提供 Claude、Cursor、Codex、Copilot、Gemini、OpenCode、Cline 页面及机器可读入口 → 提高发现性 → 多宿主；不是每个 Agent 的专用安装器。
10. **Improve UI（main 未发布）** → 先证明设计契约/运行时关系，再最多报告三项问题并为选中项写计划，严格不改源码 → 低误报审计 → 大型项目。

**可用状态与成本：** CLI 免账号、免配置、MIT 免费，但需要 Node/npm，外部 Skill 读取需要联网。正式 [v0.2.3](https://github.com/ibelick/ui-skills/releases/tag/v0.2.3) 实测 110 个 Skill、24 分类，主要修 Windows `start`；当前 main 已到 179 个 Skill、26 分类并增加 Improve UI/Agent 页面，但 package version 仍写 0.2.3，网站/main 内容不一定能由 npm CLI 找到。仓库自带仅 v0.2.3 的 5 个/main 的 6 个，其余是外部索引。[Registry](https://github.com/ibelick/ui-skills/blob/main/src/data/registry.ts) · [CLI](https://github.com/ibelick/ui-skills/blob/main/bin/ui-skills.ts)

**同类比较：** Hallmark 是一个深而一致的端到端设计 Skill，提供 Build/Audit/Redesign/Study、主题、记忆和门禁；UI Skills 是广而灵活的路由目录，可组合 Hallmark、无障碍、性能和框架专项。相对 Anthropic frontend-design，它的价值是选择和组合，代价是不同作者规则可能冲突且没有统一质量门禁。

**限制和营销边界：** `start` 容易被误解为自动路由，实际只打印 Prompt；仅支持精确分类/slug，没有语义搜索、评分、可信度、依赖解析。外部 Raw URL 会随第三方 main 变化或 404；网站更新快于 Release；收录不代表官方审计、原创或效果背书，社区仍缺 before/after 证据和统一评分。

**为何今天关注：** 事实是正式 CLI 已有 110 条，main/网站已扩到 179 条，今日 +141。推断上，Skills 数量快速增长后，“该加载哪一个”成为新痛点，熟悉的 `categories/list/get` 和多来源聚合让它很容易传播。

## 12. Graphify

**一句话定位：** 为开发者和编码 Agent 把代码、文档、数据库结构和媒体资料整理成可查询、可追溯的关系图，让用户用关系和路径查询替代反复 grep、整库塞上下文或纯向量检索。

**典型场景：** 理解陌生仓库；修改前做影响分析；给编码 Agent 提供压缩上下文；关联代码与 ADR/PDF/媒体；在提交或切分支后增量更新，并通过 MCP/JSON/HTML/图数据库共享。

### 最值得关注的 Features

1. **本地多语言代码图** → 确定性抽取定义、调用、导入、继承、SQL 表和外键 → 无需上传源码或向量库 → 隐私仓库、架构理解；不同语言深度并不完全一致。
2. **文档与媒体语义 Pass** → 从 Markdown、PDF、Office、图片、音视频转录补充概念、决策、人物和理由 → 连接“做什么”和“为什么” → 研究库。使用云模型时内容会离开设备。
3. **`EXTRACTED`/`INFERRED`/`AMBIGUOUS` 标签** → 区分源文件显式关系、解析/模型推断和歧义边，并附置信度 → 优先审查高风险连接 → Agent 引用；标签不是正确性证明。
4. **`query`/`path`/`explain`** → 本地评分与图遍历裁剪子图、找最短关系链、展示出处/社区/连接 → 快速取得小而相关的上下文 → 影响分析；它们不是生成式问答模型。
5. **社区、God Nodes 与设计理由** → 识别子系统、中心节点、意外连接、NOTE/WHY/HACK、ADR/RFC → 暴露隐性耦合 → 架构 Review。
6. **增量更新与 Git Hook** → SHA-256 增量重建，支持 `update`、`watch`、post-commit/post-checkout 和 union merge driver → 日常维护图谱而非全量重扫 → 大仓库。
7. **20+ Agent 安装目标** → 为 Claude Code、Codex、Cursor、Gemini、OpenCode、Aider、Copilot 等写入 Skill/规则/Hook → 同一图跨 Agent 使用 → 多工具团队；集成成熟度不一。
8. **HTML/JSON/Markdown 输出** → 默认交互 HTML、NetworkX JSON 和报告 → 人可浏览、Agent 可消费 → 本地探索；超过约 5,000 节点时 HTML 可能过重。
9. **多导出后端** → Wiki、Obsidian、SVG、GraphML、Mermaid/call-flow、Neo4j、FalkorDB → 进入知识管理和图数据库流程 → 团队共享。
10. **MCP 服务** → `graphifyy[mcp]` 提供 stdio/HTTP、查询、邻居、最短路径、PR 影响和分诊工具 → Agent 无需解析文件 → 自动化 Review；HTTP 对外监听需 API Key。

**可用状态与成本：** `uv tool install graphifyy` 后，基础代码解析、CLI 查询和 HTML/JSON/Markdown 免费、MIT、免账号。MCP、PDF、Office、Video、Watch、Neo4j、FalkorDB、Leiden、Terraform 等需 extras，部分需要 Python 版本或编译环境；云模型产生第三方费用，Ollama 可保持语义 Pass 本地。当前无付费 Pro，Enterprise 早期访问中的 merge-gate、Digest、Jira 等不能当作普遍可用。今日 [v0.9.17](https://github.com/Graphify-Labs/graphify/releases/tag/v0.9.17) 修复增量、缓存、隐私和解析问题，但项目仍是 Pre-1.0，`--mode deep` 的 Skill 侧传递仍待后续。[README](https://github.com/Graphify-Labs/graphify/blob/v0.9.17/README.md) · [How it works](https://github.com/Graphify-Labs/graphify/blob/v0.9.17/docs/how-it-works.md)

**同类比较：** 相比传统代码 RAG，强在显式关系、多跳、来源标签和无向量库，弱在同义改写和纯语义问题；相比 Sourcegraph Cody，强在 MIT、本地文件化、审计标签和免账号，弱在企业索引、IDE 体验和完整编码闭环；相比研究型 RepoGraph，产品入口、媒体、MCP、Hook 和增量维护更完整，但 Benchmark 任务不可直接横比。

**限制和营销边界：** 官方 Benchmark 是项目方自建小样本；“71.5× Token reduction”来自特定语料，“构图零 LLM 成本”主要指 AST-only 代码图，不含文档/媒体语义 Pass。Release 频繁修复增量和图正确性，说明 Feature 可用但仍在硬化；README/Security 文档存在局部版本同步欠缺。

**为何今天关注：** 事实是 v0.9.17 今日发布并集中修复真实增量/隐私问题，项目当日 +1,623。推断上，“query, don’t grep”、源码本地、一键接多 Agent 和可视 HTML 共同带来传播，但异常高的累计 Stars 本身不能证明生产用户规模。

## 今天最值得体验的 3 个项目

1. **[Hallmark](https://github.com/Nutlope/hallmark)**：安装成本最低，`audit`、`study`、项目记忆和组件八状态比单纯“再生成一个漂亮页面”更能改变持续开发质量。建议先对现有页面执行只读 Audit，再用 Redesign 对照。
2. **[Bonsai-demo](https://github.com/PrismML-Eng/Bonsai-demo)**：最适合验证“本地 27B 多模态 Agent”是否已进入个人硬件可用区间。优先体验视觉、可控 Thinking、OpenAI Tool Calling 和 MCP；不要一开始追求 262K 上下文或手机部署。
3. **[DeepTutor](https://github.com/HKUDS/DeepTutor)**：最完整的垂直 Feature 链，从资料、Research、Mastery、Memory 到 Interactive Book 和 Partner，适合判断长期 Agent 工作区是否比一次性问答更有价值。

> 如果你的首要问题是大型代码库上下文，而非通用体验，今日可把 Graphify 作为 DeepTutor 的替代优先项：先运行本地代码图，再用 `query/path/explain` 验证它是否真的减少 Agent 的无效搜索。

## 值得持续跟踪的新 Feature 方向

1. **Agent Operating Layer**：Agent Groups、任务状态、排程、验收、运行报告与人工介入将从“聊天附属功能”发展为独立工作操作系统。
2. **可嵌入 Agent Runtime 与 Harness 可移植性**：一边是六语言 SDK 把成熟 Runtime 嵌入应用，另一边是低成本模型适配不同 Harness；两条路线会共同推动 Agent 从产品变成基础能力。
3. **AI Observability → Eval → 实验闭环**：Trace 本身已不稀缺，下一阶段竞争点是生产样本如何进入人工 Review、自动 Eval、Prompt/模型实验和产品指标。
4. **可审计、可编辑长期记忆**：L1/L2/L3 证据链和白盒记忆说明，用户需要的不只是“系统记住我”，还要知道记住了什么、为什么、怎样删除。
5. **Agent 的业务语义标准**：Ossie 代表的方向是把指标、同义词、说明和样例问题标准化；真正价值取决于厂商原生采用与查询 Runtime，而非只发布 Schema。
6. **本地多模态工具模型**：低比特模型开始同时交付视觉、推理、Tool Calling、MCP 和 OpenAI API，后续应关注独立质量复现、长链 Agent 可靠性和主线运行时兼容。
7. **Skill 发现、评分与组合治理**：Skills 从“有没有”进入“选哪个、能否信任、是否冲突、如何评测”；UI Skills 的路由和 awesome-llm-apps 的局部门禁只是早期答案。
8. **设计 Agent 的真实视觉闭环**：Hallmark 已把结构、审计和门禁做成规则，下一步需要截图、浏览器交互、视觉回归和人类反馈形成可重复的设计 Eval。
